# Tema 15. REFACTORIZACIÓN Y DEUDA TÉCNICA: COPILOT COMO COPILOTO, NO COMO CIRUJANO CIEGO

Refactorizar código legacy es una de las tareas más arriesgadas del desarrollo. La IA comete el mismo error que el desarrollador inexperto: quiere cambiar todo de golpe. Abre un `LegacyBillingService` de 600 líneas, identifica doce problemas y propone reescribirlo entero en una sola operación. El resultado compila, pero nadie puede garantizar que se comporta igual que antes. En producción, "se comporta igual que antes" es lo único que importa.

El enfoque correcto invierte el orden: primero la red de seguridad, luego el bisturí. En este tema aprenderás a dirigir al agente como un cirujano metódico que hace una incisión cada vez, comprueba que el paciente sigue vivo, y solo entonces hace la siguiente.

Stack de referencia:

- Java 21+ (records, sealed classes, streams modernos)
- Spring Boot 3.4+
- JUnit 5, AssertJ, Mockito
- PITest (mutation testing), ArchUnit (validación arquitectónica)

## 1. Detectar Deuda: El diagnóstico asistido

### El diagnóstico antes del bisturí

El primer paso no es refactorizar — es entender qué hay que refactorizar y por qué. La IA puede analizar una clase y producir un inventario de problemas ordenado por impacto. Sin este paso, el agente genera cambios cosméticos (renombrar variables) cuando el problema real es estructural (God Object con cinco responsabilidades).

**Prompt al Agente:**
> "Analiza `LegacyBillingService.java` en `shopflow-orders` y produce un inventario de deuda técnica. Para cada problema detectado, indica: tipo de problema (God Object, magic number, acoplamiento, duplicación, etc.), localización (nombre del método, línea aproximada), impacto en mantenibilidad (alto/medio/bajo), y si es seguro refactorizarlo sin tests previos. Ordena por impacto descendente. No modifiques nada todavía."

Ejemplo de inventario de deuda que el agente debería producir para `LegacyBillingService`:

```
1. [ALTO] God Object — generateAndSendInvoice() tiene 5 responsabilidades:
   cálculo fiscal, generación PDF, persistencia JDBC, envío email, marcado de pedido.
   Localización: líneas 45-280. Seguro sin tests: NO.

2. [ALTO] Estado mutable compartido — campo estático `dailyEmailCount`:
   no thread-safe, produce condiciones de carrera bajo carga.
   Localización: línea 18. Seguro sin tests: NO.

3. [ALTO] SQL concatenado (SQL Injection potencial) — construcción manual del INSERT.
   Localización: línea 198. Seguro sin tests: SÍ (refactor a PreparedStatement sin cambiar lógica).

4. [MEDIO] Magic numbers — tipos impositivos hardcoded (0.21, 0.10, 0.04).
   Localización: líneas 89-104. Seguro sin tests: SÍ (extraer a constantes o enum).

5. [MEDIO] Duplicación — resolveRate() y la lógica inline en generateAndSendInvoice()
   calculan el IVA de forma independiente. Si cambia el tipo, hay que cambiar en dos sitios.
   Localización: líneas 89 y 156. Seguro sin tests: NO.

6. [BAJO] Error silenciado — la excepción de persistencia se captura con log.warn() y
   la ejecución continúa: la factura se "genera" aunque no se haya guardado en BD.
   Localización: línea 215. Seguro sin tests: NO.
```

### Identificar Hotspots con métricas

No toda la deuda merece el mismo esfuerzo. Los hotspots son los ficheros con más deuda Y más cambios frecuentes en git: alto coste de mantenimiento y alto riesgo de bugs. La IA puede cruzar el análisis de código con el historial de git.

**Prompt al Agente:**
> "Ejecuta `git log --oneline --name-only src/main/java/com/shopflow/ | grep '.java$' | sort | uniq -c | sort -rn | head -20` para identificar los 20 ficheros con más cambios en el último año. Para cada uno, dame una estimación de complejidad ciclomática basándote en el número de `if`, `for`, `while`, `switch` y `catch` que contiene. Los ficheros con alta frecuencia de cambio Y alta complejidad son los hotspots prioritarios."

## 2. Refactor Safe-by-Default: Pasos pequeños y Snapshots

### La regla que el agente rompe siempre

Nunca se modifica una línea de lógica antes de tener una red de seguridad. La IA incumple esta regla sistemáticamente: identifica un problema, lo "corrige" y te muestra el resultado. Sin tests, no puedes saber si el comportamiento es el mismo.

El flujo correcto tiene tres fases obligatorias:

```
1. Red de seguridad    →    2. Cambio mínimo    →    3. Verificación
   (tests de              (una extracción,           (mvn test en verde
    caracterización)       un renombrado,              antes del siguiente
                           una constante)              cambio)
```

### Tests de Caracterización: la red de seguridad

Un test de caracterización no verifica que el código es correcto — verifica que sigue comportándose igual que antes del refactor. La diferencia es sutil pero crítica: estamos describiendo el comportamiento actual, no el comportamiento deseado.

**Prompt al Agente:**
> "Genera tests de caracterización para `LegacyBillingService.generateAndSendInvoice()`. No tests unitarios normales — tests que capturen el comportamiento actual, incluso si ese comportamiento es incorrecto. Estrategia:\n1. Mockea todas las dependencias externas (JDBC, email sender, PDF generator) para aislar la clase.\n2. Para cada rama de lógica visible (tipo impositivo, descuento, tipo de cliente), crea un test que capture el valor que devuelve HOY el método.\n3. No uses `assertEquals` con el valor correcto — usa el valor que el código produce actualmente, aunque sospeches que está mal.\n4. El objetivo: si cualquier refactor cambia un valor, el test falla y te avisa."

```java
// Test de caracterización — captura el comportamiento actual, no el correcto
@ExtendWith(MockitoExtension.class)
class LegacyBillingServiceCharacterizationTest {

    @Mock private JdbcTemplate jdbcTemplate;
    @Mock private JavaMailSender mailSender;
    @Mock private PdfGeneratorStub pdfGenerator;

    @InjectMocks
    private LegacyBillingService service;

    @Test
    @DisplayName("[CARACTERIZACIÓN] IVA general para cliente B2B sin descuento")
    void characterize_TaxCalculation_B2BClientNoDiscount() {
        // Este test captura el valor que el código produce HOY
        // Si el cálculo es incorrecto, lo sabremos después del refactor
        var result = service.generateAndSendInvoice(
            buildOrder("B2B", "ESP", null),
            "customer@example.com"
        );

        // Valor capturado del comportamiento actual: 21% sobre 100€ = 21€
        // NOTA: si este valor es incorrecto en producción, es un bug previo al refactor
        assertThat(result.taxAmount()).isEqualByComparingTo("21.00");
        assertThat(result.totalWithTax()).isEqualByComparingTo("121.00");
    }

    @Test
    @DisplayName("[CARACTERIZACIÓN] IVA reducido para productos de alimentación")
    void characterize_TaxCalculation_FoodCategory() {
        var result = service.generateAndSendInvoice(
            buildOrder("B2C", "ESP", "FOOD"),
            "customer@example.com"
        );

        // Captura actual: 10% para categoría FOOD
        assertThat(result.taxAmount()).isEqualByComparingTo("10.00");
    }

    @Test
    @DisplayName("[CARACTERIZACIÓN] Email se envía siempre aunque la persistencia falle")
    void characterize_EmailSentEvenWhenPersistenceFails() {
        // Este test documenta un BUG conocido — la factura se "envía" aunque no se persista
        // No lo corriges ahora — lo corriges después de extraer las responsabilidades
        doThrow(new DataAccessException("DB down") {})
            .when(jdbcTemplate).update(anyString(), any(Object[].class));

        // El método no lanza excepción aunque la BD falle
        assertThatCode(() -> service.generateAndSendInvoice(
            buildOrder("B2B", "ESP", null), "customer@example.com"
        )).doesNotThrowAnyException();

        // Y el email se envió de todas formas
        verify(mailSender).send(any(SimpleMailMessage.class));
    }
}
```

Este último test documenta un bug: la factura se marca como enviada aunque no se haya persistido. El test **no corrige el bug** — lo documenta para poder corregirlo de forma segura después del refactor.

Indicar qué tests deben seguir en verde tras el paso.\n3. No incluir más de un tipo de cambio por paso (no 'extrae y renombra' en el mismo paso).\nNo ejecutes ningún paso todavía — solo la lista."

## 3. Migraciones Típicas: Limpieza Estructural

### Extracción de responsabilidades del God Object

`LegacyBillingService.generateAndSendInvoice()` tiene cinco responsabilidades en un método de 230 líneas. El refactor estructural las separa en componentes independientes, uno por paso.

**Paso 2 — Extracción de TaxCalculationService:**

**Prompt al Agente:**
> "Extrae la lógica de `resolveRate()` y el cálculo de impuestos inline de `LegacyBillingService` (líneas 89-130) a un nuevo `TaxCalculationService` en `com.shopflow.orders.domain.service`. El nuevo servicio expone un único método `calculate(BigDecimal subtotal, String countryCode, String category)` que devuelve un record `TaxResult(BigDecimal taxAmount, BigDecimal totalWithTax, String taxCategory)`. Sin anotaciones de Spring en el record. `TaxCalculationService` se registra como `@Bean` en `infrastructure/config/`. Después de la extracción, `LegacyBillingService` llama a `taxCalculationService.calculate()` — sin duplicar la lógica. Ejecuta `mvn test` y confirma que los tests de caracterización siguen en verde."

```java
// domain/service/TaxCalculationService.java — Java puro, sin framework
public class TaxCalculationService {

    // Constantes — eliminan los magic numbers
    private static final BigDecimal TAX_GENERAL   = new BigDecimal("0.21");
    private static final BigDecimal TAX_REDUCED   = new BigDecimal("0.10");
    private static final BigDecimal TAX_SUPER_RED = new BigDecimal("0.04");
    private static final BigDecimal TAX_ZERO      = BigDecimal.ZERO;

    public TaxResult calculate(BigDecimal subtotal, String countryCode, String category) {
        Objects.requireNonNull(subtotal, "Subtotal is required");
        Objects.requireNonNull(countryCode, "Country code is required");

        BigDecimal rate = resolveRate(countryCode, category);
        BigDecimal taxAmount = subtotal.multiply(rate).setScale(2, RoundingMode.HALF_UP);
        BigDecimal total = subtotal.add(taxAmount);
        String taxCategory = resolveCategoryName(rate);

        return new TaxResult(taxAmount, total, taxCategory);
    }

    private BigDecimal resolveRate(String countryCode, String category) {
        if ("CANARY_ISLANDS".equals(countryCode)) return TAX_ZERO;
        return switch (category) {
            case "FOOD", "MEDICINE" -> TAX_SUPER_RED;
            case "CULTURE", "HOSPITALITY" -> TAX_REDUCED;
            default -> TAX_GENERAL;
        };
    }

    private String resolveCategoryName(BigDecimal rate) {
        if (rate.compareTo(TAX_GENERAL) == 0)   return "GENERAL_21";
        if (rate.compareTo(TAX_REDUCED) == 0)   return "REDUCED_10";
        if (rate.compareTo(TAX_SUPER_RED) == 0) return "SUPER_REDUCED_4";
        return "EXEMPT_0";
    }

    public record TaxResult(
        BigDecimal taxAmount,
        BigDecimal totalWithTax,
        String taxCategory
    ) {}
}
```

## 4. Modernización Java/Spring: El "Face-lift" del Código

### De Java 8 a Java 21: qué pedir y en qué orden

La modernización del código no es una urgencia técnica si el código funciona. Es una inversión en legibilidad y mantenibilidad. Aplícala después del refactor estructural, no antes.

**Prompt al Agente:**
> "Moderniza `TaxCalculationService` para usar Java 21. Cambios específicos:\n1. Sustituye el bloque `if-else if` por `switch` expression con `->` (sin `break`).\n2. Sustituye las constantes `BigDecimal` con `String` por un `enum TaxRate` con el valor como campo.\n3. El `record TaxResult` ya usa records de Java 16+ — no hay cambio.\nNo hagas otros cambios. Ejecuta `mvn test` en verde."

Ejemplo de modernización del `switch`:

```java
// ⚠️ Antes — Java 8, verboso y propenso a olvidar el break
private BigDecimal resolveRate(String countryCode, String category) {
    if ("CANARY_ISLANDS".equals(countryCode)) {
        return BigDecimal.ZERO;
    }
    switch (category) {
        case "FOOD":
        case "MEDICINE":
            return new BigDecimal("0.04");
        case "CULTURE":
        case "HOSPITALITY":
            return new BigDecimal("0.10");
        default:
            return new BigDecimal("0.21");
    }
}

// ✅ Después — Java 21, expresivo, exhaustivo, sin break
private TaxRate resolveRate(String countryCode, String category) {
    if ("CANARY_ISLANDS".equals(countryCode)) return TaxRate.EXEMPT;
    return switch (category) {
        case "FOOD", "MEDICINE"         -> TaxRate.SUPER_REDUCED;
        case "CULTURE", "HOSPITALITY"   -> TaxRate.REDUCED;
        default                         -> TaxRate.GENERAL;
    };
}

// Enum tipado — elimina magic numbers y centraliza el valor
enum TaxRate {
    GENERAL(new BigDecimal("0.21")),
    REDUCED(new BigDecimal("0.10")),
    SUPER_REDUCED(new BigDecimal("0.04")),
    EXEMPT(BigDecimal.ZERO);

    final BigDecimal rate;
    TaxRate(BigDecimal rate) { this.rate = rate; }
}
```

### Records para DTOs y Value Objects

**Prompt al Agente:**
> "En `infrastructure/rest/dto/`, tengo varios DTOs de request/response definidos como clases con campos, getters, constructores y `toString`. Conviértelos a Java Records. Condiciones:\n1. Solo si el DTO no extiende ninguna clase.\n2. Mantén las anotaciones de Jackson (`@JsonProperty`, `@JsonInclude`) en el record.\n3. Si el DTO tiene un método de transformación (`toCommand()`), mantenlo como método de instancia en el record.\n4. Ejecuta `mvn test` en verde tras cada conversión."

```java
// Antes — DTO verboso con 40 líneas
public class CreateOrderRequest {
    @JsonProperty("customer_id")
    private String customerId;
    @JsonProperty("lines")
    private List<OrderLineRequest> lines;

    // Constructor, getters, toString... 30 líneas más
    
    public CreateOrderCommand toCommand() {
        return new CreateOrderCommand(
            UUID.fromString(customerId),
            lines.stream().map(OrderLineRequest::toCommand).toList()
        );
    }
}

// Después — record compacto con la misma funcionalidad
public record CreateOrderRequest(
    @JsonProperty("customer_id") String customerId,
    @JsonProperty("lines") List<OrderLineRequest> lines
) {
    public CreateOrderCommand toCommand() {
        return new CreateOrderCommand(
            UUID.fromString(customerId),
            lines.stream().map(OrderLineRequest::toCommand).toList()
        );
    }
}
```

## 5. Revisiones Críticas: Los efectos secundarios

La IA puede limpiar el código y romper el comportamiento en runtime de formas que los tests de compilación no detectan. Hay tres trampas que el Senior debe revisar explícitamente.

### Trampa 1: Transacciones rotas por extracción de clase

El proxy de `@Transactional` de Spring funciona interceptando llamadas externas al bean. Cuando extraes un método a una nueva clase, la transacción puede dejar de funcionar.

```java
// ⚠️ TRAMPA — llamada interna que pierde la transacción
@Service
public class InvoiceApplicationService {

    // Antes del refactor: markOrderAsBilled() era un método privado en esta misma clase
    // La llamada era interna y estaba DENTRO de la transacción de generateAndSendInvoice()

    @Transactional
    public Invoice generateAndSendInvoice(Order order, String email) {
        Invoice invoice = createInvoice(order);
        invoiceRepository.save(invoice);
        markOrderAsBilled(order.getId());  // ← ANTES: método privado, dentro de la TX
        return invoice;
    }

    // DESPUÉS del refactor incorrecto: sigue siendo privado pero llama a un método público
    // de otra clase que tiene su propia @Transactional
    // → puede haber dos transacciones separadas o ninguna dependiendo del propagation
    private void markOrderAsBilled(OrderId orderId) {
        orderService.markAsBilled(orderId);  // ← nueva clase, nueva transacción, nueva TX boundary
    }
}
```

**Prompt al Agente:**
> "Tras extraer `InvoiceRepository` y `OrderRepository` como dependencias de `InvoiceApplicationService`, verifica que las llamadas `invoiceRepository.save()` y `orderService.markAsBilled()` siguen ejecutándose en la misma transacción que `generateAndSendInvoice()`. Muestra el propagation de `@Transactional` de cada método involucrado y confirma si hay riesgo de que una operación confirme y la otra no."

### Trampa 2: Streams y rendimiento

Los Streams de Java no son siempre más eficientes que los bucles imperativos. El agente los propone sin analizar el contexto.

```java
// Antes — bucle imperativo en lista de 10 elementos: eficiente
List<InvoiceLine> result = new ArrayList<>();
for (OrderLine line : order.getLines()) {
    if (line.subtotal().amount().compareTo(MINIMUM_INVOICE_AMOUNT) > 0) {
        result.add(toInvoiceLine(line));
    }
}

// Después de modernización correcta — para listas pequeñas, equivalente
List<InvoiceLine> result = order.getLines().stream()
    .filter(line -> line.subtotal().amount().compareTo(MINIMUM_INVOICE_AMOUNT) > 0)
    .map(this::toInvoiceLine)
    .toList();

// ⚠️ PROBLEMA si la lista tiene millones de elementos (batch processing):
// parallelStream() puede parecer más rápido pero introduce condiciones de carrera
// si toInvoiceLine() tiene estado compartido (como el campo estático dailyEmailCount)
```

**Prompt al Agente:**
> "Antes de modernizar los bucles de `InvoiceApplicationService` a streams, analiza el tamaño típico de las colecciones que procesa. Si algún método procesa listas que pueden tener más de 10.000 elementos (batch processing), señálalo explícitamente y pregunta si debo usar `Stream` estándar, `parallelStream` o mantener el bucle imperativo con un iterador."

### Trampa 3: Estado mutable compartido entre threads

`LegacyBillingService` tiene un campo estático `dailyEmailCount` no sincronizado. La IA puede "limpiarlo" sin resolver el problema de concurrencia subyacente.

```java
// ⚠️ Campo estático mutable — problema de concurrencia
@Service
public class LegacyBillingService {
    private static int dailyEmailCount = 0;  // ← no thread-safe

    public void sendEmail(String to, Invoice invoice) {
        if (dailyEmailCount >= MAX_DAILY_EMAILS) {
            throw new DailyEmailLimitException();
        }
        mailSender.send(buildEmail(to, invoice));
        dailyEmailCount++;  // ← race condition bajo carga
    }
}

// ✅ Solución correcta — AtomicInteger para operaciones atómicas
@Service
public class InvoiceEmailSender {
    private final AtomicInteger dailyEmailCount = new AtomicInteger(0);

    public void send(String to, Invoice invoice) {
        int current = dailyEmailCount.getAndIncrement();
        if (current >= billingProperties.dailyEmailLimit()) {
            dailyEmailCount.decrementAndGet();  // rollback del contador
            throw new DailyEmailLimitExceededException(current);
        }
        mailSender.send(buildEmail(to, invoice));
    }

    // Llamar con @Scheduled(cron = "0 0 0 * * *") en una clase separada
    public void resetDailyCount() {
        dailyEmailCount.set(0);
    }
}
```

**Prompt al Agente:**
> "Revisa todos los campos de `InvoiceApplicationService` y sus dependencias extraídas. Detecta: 1) campos de instancia que son mutables y podrían tener problemas de concurrencia si el bean es singleton (scope por defecto de Spring). 2) Para cada campo problemático, propón si la solución es `AtomicInteger`, `synchronized`, `ThreadLocal`, o extraer el estado a una fuente de verdad externa (Redis, BD). 3) Añade un test que provoque la condición de carrera y verifique que la solución la resuelve."

✅ **Checklist de sección:**
- [ ] `@Transactional` boundaries verificados tras cualquier extracción de clase
- [ ] Streams usados solo en colecciones pequeñas o con análisis de rendimiento previo
- [ ] Campos estáticos mutables reemplazados por `AtomicInteger` o equivalente thread-safe
- [ ] Tests de concurrencia para código que se ejecutará en múltiples threads

## 6. Plugin del Curso: Analista de Deuda y Refactor Guiado

Las cinco secciones anteriores definen el método: diagnóstico antes del bisturí, tests de caracterización como red de seguridad, pasos atómicos con `mvn test` entre cada uno, y verificación explícita de los efectos secundarios en runtime. El problema es que aplicar esta disciplina en un God Object de 600 líneas requiere dos decisiones costosas: saber en qué orden atacar la deuda, y saber cuándo parar. El plugin de este tema automatiza el diagnóstico y convierte el plan de refactorización en un flujo guiado con checkpoints humanos.

El subagente `spring-debt-analyst` es la evolución del `spring-domain-analyst` del Tema 12: mientras ese analiza riqueza de dominio, este analiza deuda técnica estructural en cualquier capa. La skill `/refactor-class` es la evolución del Agent Mode del Tema 6: en lugar de pedir al agente que refactorice sin control, la skill impone el método de pasos seguros con red de seguridad previa.

### Subagente: `spring-debt-analyst`

Analiza una clase e identifica la deuda técnica estructural en siete categorías, ordenada por impacto y riesgo. No propone código — produce el plan de refactorización con el orden correcto de pasos.

Crea el fichero `.claude/agents/spring-debt-analyst.md`:

```markdown
---
name: spring-debt-analyst
description: >
  Analiza la deuda técnica estructural de una clase Spring Boot. Detecta God Objects,
  magic numbers, acoplamiento excesivo, estado mutable compartido, SQL concatenado,
  errores silenciados y violaciones de capa. Produce un inventario priorizado con el
  plan de refactorización seguro. Úsalo antes de cualquier refactor para establecer
  el diagnóstico y el orden correcto de pasos. No genera código de refactorización.
model: claude-sonnet-4-6
---

Eres un arquitecto especializado en refactorización de código legacy Java/Spring Boot.
Tu función es diagnosticar la deuda técnica y producir el plan de refactorización seguro.
No generas el código refactorizado — produces el inventario y el plan ordenado de pasos.

## Categorías de análisis

Para cada clase recibida, evalúa las siete categorías:

### 1. God Object / Single Responsibility
- Cuenta las responsabilidades distintas del método más largo de la clase.
- Una responsabilidad = un motivo de cambio (fiscal, persistencia, comunicación, presentación, validación).
- Si hay más de 2 responsabilidades en un método o más de 3 en la clase: GOD OBJECT.
- Para cada responsabilidad extra: nombre del componente a extraer y su paquete destino.

### 2. Magic Numbers y Strings
- Literales numéricos o de string en lógica de negocio (`0.21`, `"FOOD"`, `1000`).
- Para cada uno: constante o enum recomendado, paquete donde definirlo.
- Distingue entre magic numbers de negocio (tipos impositivos → enum `TaxRate`) y de configuración (límites → `@ConfigurationProperties`).

### 3. Estado Mutable Compartido
- Campos de instancia mutables en beans `@Service` / `@Component` (singletons).
- Campos estáticos mutables.
- Para cada uno: `AtomicInteger`, `synchronized`, `ThreadLocal` o extracción a fuente externa.

### 4. Acoplamiento de Capas
- Imports de `jakarta.persistence` en `domain/`.
- Imports de `com.fasterxml.jackson` en `domain/`.
- Uso directo de `JdbcTemplate` o `EntityManager` fuera de `infrastructure/persistence/`.
- `@Value` inyectados en clases de dominio.
- Para cada violación: clase a crear en la capa correcta y patrón de separación.

### 5. SQL y Persistencia Inseguros
- SQL construido con concatenación de strings (riesgo de SQL Injection).
- `ResultSet` o `JdbcTemplate` fuera de `infrastructure/persistence/`.
- Conexiones gestionadas manualmente sin `try-with-resources`.
- Para cada uno: solución con `PreparedStatement`, Spring Data o repositorio JPA.

### 6. Errores Silenciados
- `catch (Exception e) { log.warn(...); }` sin relanzar ni gestionar el error.
- Excepciones genéricas capturadas donde debería propagarse una excepción de dominio.
- Flujos de ejecución que continúan tras un error parcial sin compensación.
- Para cada uno: comportamiento correcto y test de caracterización que lo documenta.

### 7. Modernización Java/Spring
- Bucles imperables modernizables a streams (solo si la colección es pequeña).
- `@Value` dispersos consolidables en `@ConfigurationProperties`.
- POJOs convertibles a records.
- `if-else if` convertibles a `switch` expression.
- NOTA: estas mejoras tienen la prioridad más baja — solo después del refactor estructural.

## Evaluación de riesgo de refactorización

Para cada problema detectado, evalúa el riesgo:
- **SIN TESTS PREVIOS**: el comportamiento puede cambiar inadvertidamente.
- **SEGURO SIN TESTS**: el cambio es mecánico y no altera lógica (renombrar constante, añadir PreparedStatement).

## Formato de respuesta

**Resumen ejecutivo**: clase analizada, número de problemas por categoría, puntuación de deuda (0-10, donde 10 es máxima deuda).

**Inventario de deuda** (ordenado por impacto):
| # | Categoría | Descripción | Localización | Impacto | Riesgo sin tests |
|---|---|---|---|---|---|
| 1 | God Object | generateAndSendInvoice() tiene 5 responsabilidades | líneas 45-280 | ALTO | SÍ |

**Plan de refactorización** (orden seguro):
    ```
    Paso 0: Generar tests de caracterización para [métodos identificados]
    Paso 1: [Cambio seguro sin tests] → mvn test verde
    Paso 2: [Extracción mínima] → mvn test verde
    ...
    Paso N: [Modernización Java] → mvn test verde (siempre al final)
    ```

**Advertencias de efectos secundarios**:
- Transacciones afectadas por extracciones.
- Estado compartido que introduce riesgo de concurrencia.
- Comportamientos actuales que son bugs (documentar como tests de caracterización, no corregir en el refactor estructural).

**Veredicto**: ¿es seguro empezar a refactorizar? ¿qué tests de caracterización son imprescindibles antes del primer cambio?
```

## Fase 1 — Diagnóstico con `spring-debt-analyst`

Lee el fichero indicado por el usuario.

## Fase 2 — Red de Seguridad (Tests de Caracterización)

Si ya existen tests para la clase: analiza si son tests de caracterización válidos (capturan comportamiento actual) o tests de comportamiento esperado. Informa al usuario cuáles cubren qué.

Si no existen tests o la cobertura es insuficiente: genera tests de caracterización que:
1. Mockeen todas las dependencias externas (BD, email, HTTP).
2. Para cada rama lógica principal: capturen el valor que el código produce HOY.
3. Documenten explícitamente los bugs conocidos como tests (con comentario `// BUG CONOCIDO: ...`).

Ejecuta: `mvn test -pl <módulo>`

Si falla: corrige los tests de caracterización hasta que pasen. No toques la lógica de producción.

Checkpoint:
**"Tests de caracterización en verde. ¿El comportamiento capturado es el esperado? ¿Hay bugs documentados que debemos corregir ahora o después? ¿Procedo con la Fase 3? (sí/no)"**

## Fase 3 — Pasos Atómicos de Refactorización

3. Ejecuta: `mvn test -pl <módulo>`

5. Si rojo: diagnostica el fallo, propón la corrección mínima para volver a verde. No avances hasta tener verde.

Checkpoint tras cada paso:
**"[Paso N] completado. mvn test: VERDE. ¿Procedo con [Paso N+1]: [descripción]? (sí/no/parar aquí)"**

Si el usuario dice "parar aquí": haz commit del estado actual con un mensaje descriptivo y cierra la skill.

## Fase 4 — Verificación de Efectos Secundarios

Tras completar todos los pasos, verifica las tres trampas críticas:

**Transacciones:**
Muestra el grafo de llamadas de los métodos `@Transactional` antes y después del refactor. Señala si alguna extracción cambió los boundaries transaccionales.

**Estado compartido:**
Lista todos los campos de instancia de las clases resultantes. Señala los mutables en beans singleton.

**Comportamientos en runtime:**
Ejecuta: `mvn test -pl <módulo>` (suite completa, no solo los tests de caracterización).

Checkpoint final:
**"¿Los efectos secundarios identificados requieren acción antes de mergear? (sí/no)"**

## Fase 5 — Modernización (Opcional)

#### 📊 Tabla: Refactorización "Ciega" vs. Refactorización "Senior"

| Acción | Lo que la IA propone sola | Lo que el Senior le pide |
| :--- | :--- | :--- |
| **Refactorizar un God Object** | Reescribe todo de golpe — 8 clases nuevas, la original desaparece | Tests de caracterización primero, una extracción por paso, `mvn test` entre cada uno |
| **Limpiar SQL concatenado** | Cambia el String por `PreparedStatement` | Verifica si hay tests que cubren la query; si no, genera el test de integración primero |
| **Modernizar a streams** | Convierte todos los bucles sin analizar el tamaño de la colección | Analiza si la colección puede tener millones de elementos antes de proponer stream vs. bucle |
| **Resolver estado mutable** | Añade `synchronized` en el método | Analiza si `AtomicInteger` es suficiente o si el estado debe externalizarse |
| **Extraer una clase** | Mueve el código y crea el bean nuevo | Verifica los boundaries de `@Transactional` antes y después de la extracción |
| **Renombrar semántico** | Renombra en el fichero que tiene abierto | Busca todas las referencias (imports, tests, Javadoc, comentarios) y actualiza todas |
| **Eliminar código muerto** | Borra el método porque no ve llamadas | Verifica con `grep` y con el test de reflexión que no hay invocación dinámica |

#### 💡 Tip de Senior: "El Riesgo del Refactor Cosmético"

La IA mejora visualmente el código sin mejorar su arquitectura. Puede convertir todos los `for` a streams, todos los constructores a records y todos los `@Value` a `@ConfigurationProperties`, y el resultado es código Java 21 moderno que sigue siendo un God Object con cinco responsabilidades y sin tests.

La deuda técnica que realmente importa no es la deuda de sintaxis — es la deuda de diseño. Un método de 230 líneas que mezcla lógica fiscal, persistencia JDBC y envío de email es igual de costoso de mantener en Java 8 que en Java 21. La refactorización estructural (separar responsabilidades, aislar capas) reduce el coste de cambio. La modernización de sintaxis reduce la fatiga de lectura.

El orden correcto: **estructura primero, sintaxis después**. Nunca al revés.

#### 🔍 Referencias de Utilidad

- **Refactoring: Improving the Design of Existing Code (Martin Fowler, 2ª edición):** El catálogo canónico de movimientos de refactorización. Los capítulos "Extract Method", "Extract Class" y "Move Method" son los más relevantes para este tema.
- **Working Effectively with Legacy Code (Michael Feathers):** La referencia definitiva para introducir tests en código que no los tiene. El capítulo "Seams" explica cómo aislar dependencias en código legacy para poder testearlo.
- **Continuous Delivery (Jez Humble & David Farley):** Capítulo 8 sobre la estrategia de "pasos pequeños y reversibles" que fundamenta la metodología de refactorización de este tema.
- **ArchUnit Documentation** (archunit.org): Guía de configuración de reglas de arquitectura en tests de JUnit para verificar que el refactor no rompe las restricciones de capa.
- **PITest — PIT Mutation Testing** (pitest.org): Complementario al testing de caracterización — verifica que los tests añadidos durante el refactor realmente protegen la lógica extraída.
