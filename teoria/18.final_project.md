# Tema 18. PROYECTO FINAL: EL PLUGIN SHOPFLOW-SPRING

A lo largo del curso has construido, tema a tema, nueve subagentes y nueve skills dentro del directorio `.claude/` del repositorio ShopFlow. Cada uno resuelve un problema concreto: revisar la arquitectura hexagonal, generar tests con el criterio del Chaos Engineer, detectar el dual-write en Kafka, auditar vulnerabilidades OWASP. Ese conjunto de herramientas es hoy un activo del proyecto ShopFlow. El problema: está atado a ese repositorio. Para usarlo en el microservicio de Suscripciones que vas a construir en este tema, o en el próximo proyecto de tu empresa, tendrías que copiar los ficheros manualmente.

Un **plugin de Claude Code** resuelve exactamente eso: empaqueta el conjunto de subagentes y skills en un módulo portable, versionado y distribuible que se instala en cualquier proyecto con un solo comando. El proyecto final del curso tiene dos partes: primero convertir el trabajo del curso en un plugin reutilizable, y después usarlo para construir el `Servicio de Gestión de Suscripciones` de ShopFlow siguiendo el flujo completo — brief, spec, scaffolding, lógica de dominio, tests, PR.

## 1. Brief Funcional, Restricciones y Definition of Done

### El reto

Implementar el `Servicio de Gestión de Suscripciones` de ShopFlow como nuevo bounded context:

- Un `Customer` se suscribe a un `Plan` (mensual o anual).
- El sistema genera cobros periódicos vía el servicio de pagos externo `shopflow-payments`.
- Publica eventos `SubscriptionActivated`, `SubscriptionRenewed`, `SubscriptionCancelled` con Outbox Pattern.
- `shopflow-notifications` consume esos eventos y envía emails al cliente.

### Restricciones técnicas

- Java 21+, Spring Boot 3.4+, Maven 3.9+
- Arquitectura Hexagonal (misma estructura de paquetes que `shopflow-orders`)
- PostgreSQL con Testcontainers — sin H2
- Kafka con Outbox Pattern — sin `kafkaTemplate.send()` directo en `@Transactional`
- Sin `@Autowired` de campo, sin Lombok en `domain/`

### Definition of Done

```
□ mvn verify pasa sin fallos
□ ArchUnit tests pasan — sin violaciones de la arquitectura hexagonal
□ Contrato OpenAPI en docs/openapi/subscriptions-api.yaml
□ No hay @Autowired por campo ni Lombok en domain/
□ Outbox Pattern implementado (no kafkaTemplate.send() directo)
□ Se ha usado el plugin shopflow-spring durante el desarrollo
□ PR con descripción completa generada con /shopflow-spring:create-pr
□ El historial de commits refleja: brief → spec → scaffold → lógica → tests → PR
```

## 2. Qué es un Plugin de Claude Code

### Standalone vs. Plugin

Durante el curso has trabajado con la configuración **standalone**: los subagentes en `.claude/agents/` y las skills en `.claude/commands/` del repositorio ShopFlow. Es la forma correcta de empezar — iteración rápida, sin overhead de empaquetado.

Un **plugin** es el paso siguiente cuando quieres:

| Necesidad | Standalone `.claude/` | Plugin |
|---|---|---|
| Disponible solo en ShopFlow | ✅ | ❌ |
| Disponible en cualquier proyecto | ❌ | ✅ |
| Compartido con el equipo vía git | Solo si se versiona en el mismo repo | Repositorio propio, instalable |
| Actualizable en todos los proyectos a la vez | No — copia manual | `claude plugin update` |
| Namespacing para evitar conflictos | No — colisión de nombres | `/shopflow-spring:skill` |
| Distribución a la comunidad | No | Plugin marketplace |

La diferencia concreta: con standalone, `/scaffold-feature` es una skill disponible en ShopFlow. Con el plugin instalado en cualquier proyecto, `/shopflow-spring:scaffold-feature` está disponible en todos tus proyectos Spring Boot.

### Anatomía de un plugin

Un plugin es un directorio con esta estructura:

```
shopflow-spring/                    ← raíz del plugin
├── .claude-plugin/
│   └── plugin.json                 ← manifiesto: nombre, versión, autor
├── agents/                         ← subagentes especializados
│   ├── spring-boot-reviewer.md
│   ├── spring-architecture-guard.md
│   └── spring-security-auditor.md
└── commands/                       ← skills (slash commands)
    ├── scaffold-feature.md
    ├── audit-architecture.md
    └── audit-security.md
```

**Regla fundamental**: solo `plugin.json` va dentro de `.claude-plugin/`. Los directorios `agents/`, `commands/`, `hooks/` van en la **raíz del plugin**, no dentro de `.claude-plugin/`.

### Manifiesto: `plugin.json`

El fichero `.claude-plugin/plugin.json` define la identidad del plugin:

```json
{
  "name": "shopflow-spring",
  "description": "Plugin del curso IA para Desarrolladores Spring. Subagentes y skills para arquitectura hexagonal, DDD, EDA, testing, refactoring, seguridad y flujo de equipo.",
  "version": "1.0.0",
  "author": {
    "name": "Tu Nombre",
    "email": "tu@email.com",
    "url": "https://github.com/tu-usuario"
  },
  "homepage": "https://github.com/tu-usuario/shopflow-spring-plugin",
  "repository": "https://github.com/tu-usuario/shopflow-spring-plugin",
  "license": "MIT",
  "keywords": ["spring-boot", "hexagonal", "ddd", "kafka", "testing", "security"]
}
```

El campo `name` define el namespace de todas las skills del plugin. Si el nombre es `shopflow-spring`, la skill `scaffold-feature` se invoca como `/shopflow-spring:scaffold-feature`.

## 3. Construcción del Plugin Paso a Paso

### Paso 0: Preparar el entorno

Parte del estado final del curso (`v17-github`). Los ficheros que vas a empaquetar están en `shopflow/.claude/`:

```bash
git checkout v17-github

# Verifica que los ficheros del curso están presentes
ls shopflow/.claude/agents/
ls shopflow/.claude/commands/
```

Deberías ver los nueve subagentes y las nueve skills construidas a lo largo de los temas 9 a 17.

### Paso 1: Crear la estructura del plugin

Crea el repositorio del plugin fuera del repositorio ShopFlow — el plugin es un proyecto independiente:

```bash
# Fuera del directorio shopflow/
cd ..
mkdir shopflow-spring-plugin
cd shopflow-spring-plugin

# Crear el directorio del manifiesto
mkdir .claude-plugin

# Crear los directorios de contenido en la RAÍZ (no dentro de .claude-plugin/)
mkdir agents
mkdir commands
```

### Paso 2: Crear el manifiesto

```bash
cat > .claude-plugin/plugin.json << 'EOF'
{
  "name": "shopflow-spring",
  "description": "Subagentes y skills para proyectos Spring Boot con arquitectura hexagonal, DDD, EDA, testing, refactoring, seguridad y flujo de equipo.",
  "version": "1.0.0",
  "author": {
    "name": "Tu Nombre",
    "url": "https://github.com/tu-usuario"
  },
  "homepage": "https://github.com/tu-usuario/shopflow-spring-plugin",
  "repository": "https://github.com/tu-usuario/shopflow-spring-plugin",
  "license": "MIT",
  "keywords": [
    "spring-boot", "hexagonal-architecture", "ddd", "kafka",
    "testing", "security", "refactoring", "code-review"
  ]
}
EOF
```

### Paso 3: Migrar los subagentes

Copia los nueve subagentes del directorio `.claude/agents/` de ShopFlow al directorio `agents/` del plugin:

```bash
# Desde el directorio shopflow-spring-plugin/
cp ../shopflow/.claude/agents/spring-boot-reviewer.md         agents/
cp ../shopflow/.claude/agents/spring-resilience-reviewer.md   agents/
cp ../shopflow/.claude/agents/spring-architecture-guard.md    agents/
cp ../shopflow/.claude/agents/spring-domain-analyst.md        agents/
cp ../shopflow/.claude/agents/spring-event-reviewer.md        agents/
cp ../shopflow/.claude/agents/spring-test-strategist.md       agents/
cp ../shopflow/.claude/agents/spring-debt-analyst.md          agents/
cp ../shopflow/.claude/agents/spring-security-auditor.md      agents/
cp ../shopflow/.claude/agents/spring-pr-reviewer.md           agents/
```

Verifica que los subagentes tienen el frontmatter correcto. Cada fichero debe empezar con:

```markdown
---
name: spring-boot-reviewer
description: >
  [descripción de cuándo invocarlo — es lo que Claude lee para decidir si lo usa]
model: claude-sonnet-4-6
---
```

El campo `description` es crítico: Claude lo lee para decidir cuándo invocar el subagente automáticamente. Una descripción vaga hace que el subagente nunca se invoque automáticamente.

### Paso 4: Migrar las skills

Copia las nueve skills del directorio `.claude/commands/` al directorio `commands/` del plugin:

```bash
cp ../shopflow/.claude/commands/scaffold-feature.md   commands/
cp ../shopflow/.claude/commands/add-resilience.md     commands/
cp ../shopflow/.claude/commands/audit-architecture.md commands/
cp ../shopflow/.claude/commands/enrich-domain.md      commands/
cp ../shopflow/.claude/commands/add-event.md          commands/
cp ../shopflow/.claude/commands/generate-tests.md     commands/
cp ../shopflow/.claude/commands/refactor-class.md     commands/
cp ../shopflow/.claude/commands/audit-security.md     commands/
cp ../shopflow/.claude/commands/create-pr.md          commands/
```

### Paso 5: Adaptar referencias internas

Algunos ficheros de skills y subagentes hacen referencia a otros por nombre sin namespace. En el plugin, los subagentes que se invocan desde una skill deben referenciarse con el nombre exacto tal como aparece en el frontmatter del fichero del agente (sin namespace en las invocaciones internas).

Verifica que las skills que invocan subagentes usan el nombre correcto:

```bash
# Busca referencias a subagentes en las skills
grep -r "spring-" commands/ | grep -v "^Binary"
```

Por ejemplo, `/audit-security` invoca a `spring-security-auditor`. El nombre debe coincidir exactamente con el campo `name` en el frontmatter de `agents/spring-security-auditor.md`.

## 4. Distribución: GitHub como Marketplace del Equipo

### Publicar el plugin en GitHub

El plugin es un repositorio git independiente. El hosting en GitHub permite al equipo instalarlo y actualizarlo con comandos estándar.

**Paso 1: Inicializar el repositorio**

```bash
cd shopflow-spring-plugin

git init
git add .
git commit -m "feat: initial release v1.0.0

Plugin shopflow-spring con 9 subagentes y 9 skills para desarrollo
Spring Boot con arquitectura hexagonal, DDD, EDA, testing, refactoring,
seguridad y flujo de equipo.

Desarrollado en el curso IA para Desarrolladores Spring."
```

**Paso 2: Crear el repositorio en GitHub y publicar**

```bash
# Con gh CLI
gh repo create shopflow-spring-plugin --public --description "Plugin Claude Code para proyectos Spring Boot"
git remote add origin https://github.com/tu-usuario/shopflow-spring-plugin.git
git push -u origin main

# Crear el tag de la primera versión
git tag v1.0.0
git push origin v1.0.0
```

### Crear el marketplace del equipo

Para que el plugin se pueda instalar con `claude plugin install`, necesitas un fichero `marketplace.json` que actúa como índice del marketplace. Puedes alojarlo en el mismo repositorio o en uno dedicado.

Crea `marketplace.json` en la raíz del repositorio:

```json
{
  "name": "shopflow-spring-marketplace",
  "description": "Plugins del equipo ShopFlow para desarrollo con Claude Code",
  "plugins": [
    {
      "name": "shopflow-spring",
      "description": "Subagentes y skills para Spring Boot con arquitectura hexagonal, DDD, EDA, testing, refactoring, seguridad y flujo de equipo.",
      "version": "1.0.0",
      "source": ".",
      "homepage": "https://github.com/tu-usuario/shopflow-spring-plugin"
    }
  ]
}
```

Haz commit y push del `marketplace.json`:

```bash
git add marketplace.json
git commit -m "feat: add marketplace index"
git push
```

### Instalar el plugin desde GitHub

Con el marketplace publicado en GitHub, cualquier miembro del equipo puede instalar el plugin:

```bash
# Instalar en el scope de usuario (disponible en todos los proyectos)
claude plugin install shopflow-spring@https://raw.githubusercontent.com/tu-usuario/shopflow-spring-plugin/main/marketplace.json

# Instalar en el scope del proyecto (disponible para todo el equipo que clone el repo)
claude plugin install shopflow-spring@https://raw.githubusercontent.com/tu-usuario/shopflow-spring-plugin/main/marketplace.json --scope project

# Verificar que se instaló correctamente
claude plugin list
```

### Configuración de scope para el equipo

El scope `project` es el recomendado para equipos: añade la referencia al plugin en `.claude/settings.json` del repositorio, que se versiona junto al código. Cualquier desarrollador que clone el repositorio tendrá el plugin disponible:

```json
// .claude/settings.json (se versiona en git)
{
  "enabledPlugins": [
    "shopflow-spring@https://raw.githubusercontent.com/tu-usuario/shopflow-spring-plugin/main/marketplace.json"
  ]
}
```

Los tres scopes disponibles y cuándo usar cada uno:

| Scope | Fichero | Cuándo usarlo |
|---|---|---|
| `user` | `~/.claude/settings.json` | El plugin es útil para todos tus proyectos personales |
| `project` | `.claude/settings.json` (versionado) | El equipo debe tener el mismo plugin en el repositorio |
| `local` | `.claude/settings.local.json` (gitignored) | Pruebas locales sin afectar al equipo |

### Actualizar el plugin

Cuando el plugin evoluciona (nuevos subagentes, correcciones, mejoras):

```bash
# En el repositorio del plugin: actualizar la versión
# Editar .claude-plugin/plugin.json: "version": "1.1.0"
git add .
git commit -m "feat: add spring-query-analyst agent (v1.1.0)"
git tag v1.1.0
git push && git push origin v1.1.0

# En los proyectos que usan el plugin:
claude plugin update shopflow-spring
```

⚠️ **Importante**: si cambias el código del plugin sin actualizar la versión en `plugin.json`, los usuarios no recibirán la actualización debido al sistema de caché. Siempre incrementa la versión antes de publicar cambios.

## 5. Ejecución del Proyecto Final con el Plugin

### El flujo recomendado: brief → spec → scaffold → lógica → tests → PR

Con el plugin instalado, el flujo del proyecto final sigue la metodología del curso de principio a fin. Cada fase usa el modelo adecuado y la herramienta adecuada.

```bash
# Punto de partida: estado final del curso
git checkout v17-github
git checkout -b proyecto-final/subscriptions
```

**Fase 1 — Plan con Chat (tier alto, antes de tocar código)**

**Prompt al Agente:**
> "Actúa como arquitecto Java. Analiza este brief del Servicio de Gestión de Suscripciones de ShopFlow y diseña un plan de 5 fases para implementarlo. Identifica: 1) Los agregados de dominio y sus relaciones, 2) Los puertos de entrada y salida necesarios, 3) Los eventos de dominio e integración, 4) Los riesgos en la consistencia de datos con el Outbox Pattern, 5) Las dependencias con los módulos existentes (`shopflow-orders`, `shopflow-notifications`). No generes código todavía — solo el plan."

El resultado esperado es un plan estructurado como este:

```
Bounded context: subscriptions
Agregados: Subscription (root), SubscriptionPlan
Puertos de entrada: ActivateSubscriptionUseCase, RenewSubscriptionUseCase, CancelSubscriptionUseCase
Puertos de salida: SubscriptionRepository, PaymentPort, OutboxEventRepository
Eventos de dominio: SubscriptionActivated, SubscriptionRenewed, SubscriptionCancelled
Eventos de integración: SubscriptionActivatedIntegrationEvent (→ shopflow-notifications)
Riesgo de consistencia: el cargo al servicio de pagos y el evento de dominio deben ser atómicos → Outbox obligatorio
```

**Fase 2 — Spec-Driven Development: OpenAPI primero**

Antes de escribir la primera clase Java, define el contrato HTTP:

**Prompt al Agente:**
> "Genera el contrato OpenAPI 3.1 para el Servicio de Suscripciones de ShopFlow. Endpoints:\n- POST /subscriptions — suscribir un cliente a un plan\n- GET /subscriptions/{subscriptionId} — obtener el estado de una suscripción\n- DELETE /subscriptions/{subscriptionId} — cancelar una suscripción\nIncluye los schemas de request y response con todos los campos necesarios. Guarda el fichero en `shopflow-subscriptions/src/main/resources/openapi/subscriptions-api.yaml`."

**Fase 3 — Scaffolding con el Plugin**

Con el plan validado y el contrato OpenAPI en mano, usa la skill del plugin:

```bash
/shopflow-spring:scaffold-feature
```

La skill pedirá:
- Nombre de la feature: `SubscriptionActivation`
- Módulo destino: `shopflow-subscriptions`
- Descripción del caso de uso: activar una suscripción de cliente a un plan

El scaffolding genera la estructura hexagonal completa: puerto de entrada, comando, servicio de aplicación vacío, puerto de salida, y la entidad de dominio `Subscription` con el factory method `create()`.

**Fase 4 — Lógica de Dominio con Chat**

Con el scaffold en su sitio, implementa las invariantes de dominio:

**Prompt al Agente:**
> "Implementa el método `Subscription.renew()` en `shopflow-subscriptions/domain/model/Subscription.java`. Reglas de negocio:\n1. Solo se puede renovar una suscripción en estado ACTIVE.\n2. La fecha de renovación debe ser posterior a la fecha actual de expiración.\n3. El plan no puede cambiar durante la renovación (se renueva el mismo plan).\nLanza excepciones de dominio específicas para cada regla violada. Genera el test unitario de `Subscription.renew()` que falle cuando se borra la lógica."

**Fase 5 — Outbox Pattern con el Plugin**

Para añadir el evento `SubscriptionActivated` con todas las garantías:

```bash
/shopflow-spring:add-event
```

La skill seguirá el flujo del Tema 13: evento de dominio, evento de integración, entrada en el Outbox, consumidor en `shopflow-notifications`, tests de integración con Testcontainers Kafka.

**Fase 6 — Tests con el Plugin**

Para generar los tests que faltan:

```bash
/shopflow-spring:generate-tests shopflow-subscriptions/src/main/java/.../SubscriptionApplicationService.java
```

La skill analizará el servicio, identificará los casos no cubiertos, y generará los tests con el criterio del Chaos Engineer integrado.

**Fase 7 — Auditoría con el Plugin antes del merge**

```bash
# Auditoría de arquitectura
/shopflow-spring:audit-architecture

# Auditoría de seguridad si hay endpoints nuevos
/shopflow-spring:audit-security

# Crear la PR
/shopflow-spring:create-pr
```

## 6. Review Final: El Checklist del Senior

### Arquitectura

**Prompt al Agente (usar `spring-architecture-guard`):**
> "Revisa el módulo `shopflow-subscriptions` y verifica: 1) Ninguna clase de `domain/` importa de `infrastructure/`. 2) Los puertos de salida (`domain/port/out/`) son interfaces Java puras sin anotaciones de framework. 3) El servicio de aplicación solo orquesta — no contiene lógica de negocio. 4) Los tests de dominio no usan Spring."

### Consistencia de datos

**Prompt al Agente:**
> "Verifica que `SubscriptionApplicationService.activate()` usa el Outbox Pattern. Comprueba: 1) `subscriptionRepository.save()` y `outboxEventRepository.save()` están en el mismo `@Transactional`. 2) `kafkaTemplate.send()` no aparece en ningún método `@Transactional`. 3) El `OutboxRelayJob` usa `get(2, SECONDS)` con timeout explícito."

### Tests

**Prompt al Agente (usar `spring-test-strategist`):**
> "Analiza la suite de tests de `shopflow-subscriptions`. Clasifica cada test como VÁLIDO, DÉBIL o FALSO. Identifica los edge cases no cubiertos en la política de renovación y cancelación."

### Seguridad

**Prompt al Agente (usar `spring-security-auditor`):**
> "Audita los endpoints de `shopflow-subscriptions` contra el OWASP Top 10. Verifica especialmente: 1) IDOR en GET /subscriptions/{subscriptionId} — ¿se verifica que la suscripción pertenece al usuario autenticado? 2) Validación de input en POST /subscriptions — ¿los campos tienen constraints estrictas?"

#### 💡 Tip de Senior: "El Plugin como Estándar del Equipo"

El plugin `shopflow-spring` no es solo un conjunto de herramientas — es la codificación de los estándares del equipo. Cuando un nuevo desarrollador se incorpora al equipo y ejecuta `claude plugin install shopflow-spring`, en segundos tiene acceso a:

- Las reglas de arquitectura del equipo (en `spring-architecture-guard`)
- El criterio de revisión de PRs (en `spring-pr-reviewer`)
- Los estándares de seguridad (en `spring-security-auditor`)
- El método de refactorización del equipo (en `spring-debt-analyst`)

Esto transforma el plugin de una herramienta de productividad individual a un **mecanismo de transferencia de conocimiento del equipo**. Lo que antes requería meses de mentoring para que un Junior interiorizara los estándares, ahora está codificado en el plugin y disponible desde el primer día.

La siguiente evolución natural: un hook `PostToolUse` en el plugin que invoca `spring-architecture-guard` automáticamente después de cada operación de escritura de ficheros Java, bloqueando el commit si detecta una violación de capa.
