# Tema 16. SEGURIDAD CON IA EN SPRING: CÓMO PEDIR Y CÓMO VERIFICAR

La seguridad no es una capa que se añade al final — es una dimensión que se diseña desde el primer prompt. El problema con la IA en seguridad no es que genere código inseguro a propósito: es que genera código que funciona correctamente para el caso de uso legítimo y no considera los casos de abuso. Un endpoint de login generado por Copilot acepta las credenciales correctas y rechaza las incorrectas. Lo que no hace: limitar intentos fallidos, validar el formato de la entrada, proteger contra timing attacks en la comparación de contraseñas, ni devolver el mismo tiempo de respuesta si el usuario no existe.

En este tema aprenderás a usar la IA como primer detector de vulnerabilidades y como implementador de defensas, con el Senior como árbitro final: la IA puede detectar el 80% de los problemas obvios, pero el 20% restante — la lógica de negocio insegura, los permisos mal modelados, las decisiones criptográficas — requiere juicio humano que ningún modelo puede sustituir.

Stack de referencia:

- Java 21+
- Spring Boot 3.4+ con Spring Security 6
- OWASP Top 10 (2021)
- JWT (io.jsonwebtoken:jjwt)
- HashiCorp Vault / Spring Cloud Vault

## 1. Threat Modeling ligero para features: Qué pedir a la IA

### Pensar como atacante antes de escribir código

El Threat Modeling tradicional requiere dias de talleres y diagramas de flujo de datos. La versión ligera para el día a día de un equipo de desarrollo toma 15 minutos: antes de implementar una feature, usas la IA para anticipar los vectores de ataque más probables.

El prompt correcto no pide "comprueba si este código es seguro" — pide al agente que adopte el rol del atacante y enumere los vectores antes de que exista el código.

**Prompt al Agente:**
> "Voy a implementar un endpoint en `shopflow-orders` que permite a un cliente descargar su factura en PDF dado un `invoiceId`. El endpoint será GET /invoices/{invoiceId}/pdf. Antes de escribir una línea de código, actúa como un atacante especializado en APIs REST y enumera los 5 vectores de ataque más probables para este endpoint. Para cada vector: nombre del ataque, descripción en una frase, y la contramedida específica en Spring Boot que lo previene."

Ejemplo del análisis esperado:

```
Vector 1 — Insecure Direct Object Reference (IDOR)
El atacante enumera invoiceId (UUIDs o números secuenciales) para descargar
facturas de otros clientes.
Contramedida: @PreAuthorize verificando que invoice.customerId == autenticatedUserId.

Vector 2 — Path Traversal
Si invoiceId se usa para construir la ruta del PDF en disco, un valor como
"../../etc/passwd" podría filtrar ficheros del sistema.
Contramedida: validar invoiceId contra un patrón estricto (UUID), nunca concatenar
directamente a rutas del sistema de ficheros.

Vector 3 — Fuerza Bruta en descarga de PDFs
Un atacante puede iterar sobre invoiceId para descargar todas las facturas del sistema.
Contramedida: rate limiting por usuario (Bucket4j o Spring's RateLimiter).

Vector 4 — Exposición de metadatos en cabeceras
La respuesta puede filtrar rutas del sistema de ficheros en la cabecera Content-Disposition.
Contramedida: construir el nombre del fichero desde datos de la aplicación, no del sistema.

Vector 5 — Denegación de servicio por PDFs grandes
Un atacante autenticado puede generar miles de descargas de PDFs grandes simultáneamente.
Contramedida: límite de tamaño de respuesta y caching del PDF generado.
```

### Clasificación de activos y PII

Antes de diseñar los controles de seguridad, hay que saber qué datos maneja la feature. La clasificación de activos determina qué nivel de protección aplica.

**Prompt al Agente:**
> "Dado el dominio de ShopFlow (pedidos B2B, facturas, datos de clientes), clasifica los datos que maneja el sistema en tres niveles: CONFIDENCIAL (PII, datos financieros, secretos de negocio), INTERNO (datos de operación no públicos), PÚBLICO (catálogo de productos, precios de lista). Para cada nivel, indica: qué datos incluye, si requiere cifrado en reposo, si puede aparecer en logs, y si puede aparecer en mensajes de error al cliente."

### Casos de abuso explícitos

Los casos de uso describen lo que el sistema debe hacer. Los casos de abuso describen lo que el sistema NO debe permitir. La IA es especialmente útil para generar casos de abuso sistemáticamente.

**Prompt al Agente:**
> "Para el flujo de recuperación de contraseña de ShopFlow (el usuario introduce su email, recibe un token por email, usa el token para establecer una nueva contraseña), genera 5 casos de abuso que un atacante podría explotar. Para cada caso: nombre del ataque, cómo funciona, y la restricción técnica que lo previene en Spring Security."

✅ **Checklist de sección:**
- [ ] Threat modeling ejecutado antes de implementar cualquier feature con datos sensibles
- [ ] Activos clasificados por nivel: CONFIDENCIAL / INTERNO / PÚBLICO
- [ ] Al menos 3 casos de abuso generados por la IA y revisados por el Senior
- [ ] Contramedidas específicas identificadas para cada vector antes de escribir código

## 2. Validación de Inputs, AuthN/AuthZ y riesgos OWASP

### Las cuatro vulnerabilidades del código "que funciona"

El proyecto ShopFlow en la rama `exercise/topic-16` incluye `InsecureOrderController.java` — un controlador que funciona perfectamente para el caso legítimo y tiene cuatro vulnerabilidades intencionadas. La demo del tema consiste en detectarlas con la IA antes de leer el código.

```
Prompt al Chat (sin compartir el código todavía):
"Actúa como auditor de seguridad especializado en OWASP Top 10.
Voy a mostrarte un controlador REST de Spring Boot. Lista las vulnerabilidades
críticas que encuentres, con: nombre del vector, línea o sección afectada,
impacto potencial en producción, y corrección mínima en Spring Boot."
```

Las cuatro vulnerabilidades del controlador:

**Vulnerabilidad #1: Secreto hardcodeado**

```java
// ⚠️ VULNERABILIDAD: secreto hardcodeado en el código fuente
// Cualquiera con acceso al repositorio tiene la clave de API
private static final String INTERNAL_API_KEY = "sk-shopflow-prod-2024-secret-key";

@GetMapping("/internal/stats")
public ResponseEntity<OrderStats> getStats(
        @RequestHeader("X-Api-Key") String apiKey) {
    // ⚠️ Comparación en texto plano — vulnerable a timing attack
    if (!INTERNAL_API_KEY.equals(apiKey)) {
        return ResponseEntity.status(403).build();
    }
    return ResponseEntity.ok(orderService.getStats());
}
```

```java
// ✅ Corrección: secreto en @ConfigurationProperties + comparación constante en tiempo
@Value("${shopflow.internal.api-key}")
private String internalApiKey;

@GetMapping("/internal/stats")
public ResponseEntity<OrderStats> getStats(@RequestHeader("X-Api-Key") String apiKey) {
    // MessageDigest.isEqual() garantiza tiempo constante — sin timing attack
    if (!MessageDigest.isEqual(
            internalApiKey.getBytes(StandardCharsets.UTF_8),
            apiKey.getBytes(StandardCharsets.UTF_8))) {
        return ResponseEntity.status(403).build();
    }
    return ResponseEntity.ok(orderService.getStats());
}
```

**Vulnerabilidad #2: SQL Injection**

```java
// ⚠️ VULNERABILIDAD: SQL construido con concatenación de strings
// Input malicioso: customerId = "1 OR 1=1 --" → devuelve TODOS los pedidos
@GetMapping("/orders/search")
public List<Order> searchOrders(@RequestParam String customerId) {
    String rawSql = "SELECT * FROM orders WHERE customer_id = '" + customerId + "'";
    // En un contexto real: return jdbcTemplate.queryForList(rawSql);
    return orderService.searchByRawQuery(rawSql);
}
```

```java
// ✅ Corrección: Spring Data JPA con parámetros bindeados
@GetMapping("/orders/search")
public List<OrderResponse> searchOrders(
        @RequestParam @Pattern(regexp = "^[0-9a-fA-F-]{36}$") String customerId) {
    return orderService.findByCustomerId(UUID.fromString(customerId))
        .stream().map(OrderResponse::from).toList();
}

// En el repositorio — el parámetro bindeado previene SQL Injection
@Query("SELECT o FROM Order o WHERE o.customerId = :customerId")
List<Order> findByCustomerId(@Param("customerId") UUID customerId);
```

**Vulnerabilidad #3: Endpoint sin autorización**

```java
// ⚠️ VULNERABILIDAD: endpoint de borrado accesible sin autenticación ni autorización
// Cualquier cliente HTTP puede eliminar cualquier pedido
@DeleteMapping("/orders/{orderId}")
public ResponseEntity<Void> deleteOrder(@PathVariable UUID orderId) {
    orderService.delete(orderId);
    return ResponseEntity.noContent().build();
}

// ⚠️ VULNERABILIDAD adicional: endpoint admin sin restricción de rol
@GetMapping("/admin/orders/all")
public List<Order> getAllOrders() {
    return orderService.findAll();
}
```

```java
// ✅ Corrección: @PreAuthorize en el servicio, no en el controller
// (el servicio es el único punto de control — el controller puede cambiar, el servicio no)
@Service
public class OrderApplicationService {

    @PreAuthorize("hasRole('ADMIN')")
    public void delete(UUID orderId) {
        orderRepository.findById(new OrderId(orderId))
            .ifPresent(order -> orderRepository.delete(order));
    }

    @PreAuthorize("hasAnyRole('ADMIN', 'SUPPORT')")
    public List<Order> findAll() {
        return orderRepository.findAll();
    }
}
```

**Vulnerabilidad #4: Stack trace expuesto al cliente**

```java
// ⚠️ VULNERABILIDAD: e.getMessage() puede exponer rutas internas, nombres de clases
// y detalles de implementación que el atacante usará para refinar su ataque
@GetMapping("/orders/{orderId}")
public ResponseEntity<Order> getOrder(@PathVariable UUID orderId) {
    try {
        return ResponseEntity.ok(orderService.findById(orderId));
    } catch (Exception e) {
        return ResponseEntity.status(500)
            .body(new ErrorResponse(e.getMessage()));  // ← expone detalles internos
    }
}
```

```java
// ✅ Corrección: @ControllerAdvice centralizado con mensajes genéricos para el cliente
// y log detallado interno para el equipo de operaciones
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(OrderNotFoundException e) {
        log.warn("Order not found: {}", e.getMessage());
        return ResponseEntity.status(404)
            .body(new ErrorResponse("ORDER_NOT_FOUND", "The requested order does not exist"));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception e, HttpServletRequest request) {
        // Log completo internamente — el traceId permite correlacionarlo con la request
        log.error("Unexpected error on {} {}", request.getMethod(), request.getRequestURI(), e);
        // Al cliente: mensaje genérico sin detalles internos
        return ResponseEntity.status(500)
            .body(new ErrorResponse("INTERNAL_ERROR", "An unexpected error occurred"));
    }
}
```

**Prompt al Agente:**
> "Aplica las 4 correcciones a `InsecureOrderController.java` en `shopflow-orders`. Orden de cambios:\n1. Mueve el secreto `INTERNAL_API_KEY` a `@Value(\"${shopflow.internal.api-key}\")` y usa `MessageDigest.isEqual()` para la comparación.\n2. Reemplaza el SQL concatenado con una query de Spring Data JPA con parámetro bindeado. Añade `@Pattern` de validación al `@RequestParam`.\n3. Añade `@PreAuthorize(\"hasRole('ADMIN')\")` en el método `delete()` del servicio (no en el controller).\n4. Extrae el manejo de excepciones a un `@RestControllerAdvice` que devuelva mensajes genéricos al cliente y haga `log.error()` del stack trace interno.\nUn cambio por commit, `mvn test` verde entre cada cambio."

### Configuración de Spring Security 6

**Prompt al Agente:**
> "Configura Spring Security 6 para ShopFlow. JWT stateless (sin sesión). Reglas de acceso:\n- GET /api/v1/orders/** permite ROLE_USER y ROLE_ADMIN.\n- POST/PUT/DELETE /api/v1/orders/** solo ROLE_ADMIN.\n- GET /actuator/health público (sin autenticación).\n- Cualquier otra ruta requiere autenticación.\nUsar `SecurityFilterChain`, no `WebSecurityConfigurerAdapter` (deprecado). Sin CSRF (API stateless). Devuelve 401 para no autenticados, 403 para sin permisos. Sin exposición de información en la respuesta de error."

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity(prePostEnabled = true)
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtFilter;

    public SecurityConfig(JwtAuthenticationFilter jwtFilter) {
        this.jwtFilter = jwtFilter;
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)           // stateless API — no CSRF needed
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(HttpMethod.GET, "/actuator/health").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/v1/orders/**")
                    .hasAnyRole("USER", "ADMIN")
                .requestMatchers(HttpMethod.POST, "/api/v1/orders/**").hasRole("ADMIN")
                .requestMatchers(HttpMethod.PUT, "/api/v1/orders/**").hasRole("ADMIN")
                .requestMatchers(HttpMethod.DELETE, "/api/v1/orders/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .exceptionHandling(ex -> ex
                // 401 para no autenticados — sin redirigir a /login (API stateless)
                .authenticationEntryPoint((req, res, e) ->
                    res.sendError(HttpServletResponse.SC_UNAUTHORIZED))
                // 403 para sin permisos — sin detalle en el body
                .accessDeniedHandler((req, res, e) ->
                    res.sendError(HttpServletResponse.SC_FORBIDDEN))
            )
            .build();
    }
}
```

### Validación de inputs con Bean Validation

La defensa más efectiva contra inyección y abuso de parámetros es rechazar inputs que no cumplen el formato esperado antes de que lleguen a la lógica de negocio.

```java
// ✅ Validación estricta con anotaciones JSR-303
public record CreateOrderRequest(
    @NotNull
    @Pattern(regexp = "^[0-9a-fA-F-]{36}$", message = "Invalid UUID format")
    String customerId,

    @NotEmpty(message = "Order must have at least one line")
    @Size(max = 50, message = "Order cannot have more than 50 lines")
    List<@Valid OrderLineRequest> lines
) {}

public record OrderLineRequest(
    @NotBlank
    @Size(min = 3, max = 200, message = "Product name must be between 3 and 200 characters")
    String productName,

    @Positive(message = "Quantity must be positive")
    @Max(value = 9999, message = "Quantity cannot exceed 9999")
    int quantity,

    @NotNull
    @DecimalMin(value = "0.01", message = "Unit price must be greater than 0")
    @DecimalMax(value = "99999.99", message = "Unit price cannot exceed 99999.99")
    BigDecimal unitPrice
) {}
```

**Prompt al Agente:**
> "Añade validación Bean Validation (JSR-303) a todos los records de request de `shopflow-orders`. Reglas:\n1. UUIDs: `@Pattern(regexp = \"^[0-9a-fA-F-]{36}$\")`.\n2. Strings de texto libre: `@Size(min=1, max=500)` y `@NotBlank`.\n3. Cantidades monetarias: `@DecimalMin(\"0.01\")` y tipo `BigDecimal` (no `double`).\n4. Listas: `@NotEmpty` y `@Size(max=50)` con `@Valid` en cada elemento.\nConfirm que el controller tiene `@Valid` en los parámetros de método. Añade un test `@WebMvcTest` que verifique que un request con `customerId` nulo devuelve 400."

✅ **Checklist de sección:**
- [ ] Ningún secreto en el código fuente — todos en `application.yml` con perfil o en Vault
- [ ] SQL construido siempre con parámetros bindeados (Spring Data o `PreparedStatement`)
- [ ] `@PreAuthorize` en el servicio de aplicación, no solo en el controller
- [ ] `@RestControllerAdvice` centralizado — sin `e.getMessage()` en responses al cliente
- [ ] `@Valid` en todos los parámetros de request, con constraints específicas por campo

## 3. Gestión de Secretos: Higiene en Prompts y Código

### La regla de oro de la IA y los secretos

Nunca incluyas claves reales, tokens, contraseñas o certificados en un prompt. Los prompts enviados a Copilot o Claude se transmiten a servicios externos. Una clave real en un prompt es una clave comprometida.

```
// ⚠️ NUNCA hagas esto en el chat
"La clave de producción de mi base de datos es postgres://admin:P@ssw0rd123@prod-db.shopflow.com:5432/orders
Genera el application.properties con esta configuración."

// ✅ Usa placeholders siempre
"Genera el application.properties para conectar con PostgreSQL en producción.
Usa placeholder ${DB_URL}, ${DB_USERNAME}, ${DB_PASSWORD} para las credenciales.
Las credenciales reales vendrán de variables de entorno en el despliegue."
```

### Detección de secretos hardcodeados con la IA

El agente puede escanear el código en busca de patrones típicos de credenciales hardcodeadas. Es útil como segunda revisión antes de cada commit, especialmente cuando el código fue generado por la IA (que tiende a hardcodear valores de ejemplo).

**Prompt al Agente:**
> "Escanea todos los ficheros Java y YAML de `shopflow-orders` en busca de secretos hardcodeados. Patrones a buscar: 1) Strings con formato de clave API (`sk-`, `Bearer `, `token =`), 2) contraseñas en URLs JDBC (`jdbc:postgresql://user:pass@`), 3) strings que contengan `password`, `secret`, `key`, `token` asignados a literales (no a `${...}` placeholders), 4) claves privadas PEM (`-----BEGIN`). Para cada hallazgo: fichero, línea, tipo de secreto, y cómo externalizarlo."

### Configuración segura con Spring Cloud Vault

Para equipos que ya gestionan secretos en HashiCorp Vault, Spring Cloud Vault inyecta las credenciales directamente en el contexto de Spring sin pasar por variables de entorno o ficheros de configuración en disco.

**Prompt al Agente:**
> "Configura Spring Cloud Vault en `shopflow-orders` para leer los secretos del path `secret/shopflow/orders`. Los secretos que necesita el servicio son: `database.url`, `database.username`, `database.password`, y `shopflow.internal.api-key`. Usa autenticación con AppRole (no token estático). La configuración del Vault debe estar en `bootstrap.yml` (no en `application.yml`). Genera el `bootstrap.yml` y la dependencia Maven necesaria."

```yaml
# bootstrap.yml — configuración de Vault (no en application.yml)
spring:
  cloud:
    vault:
      host: vault.shopflow.internal
      port: 8200
      scheme: https
      authentication: APPROLE
      app-role:
        role-id: ${VAULT_ROLE_ID}          # variable de entorno en el pod
        secret-id: ${VAULT_SECRET_ID}      # variable de entorno en el pod
      kv:
        enabled: true
        default-context: shopflow/orders
        backend: secret
```

### Rotación de secretos sin downtime

**Prompt al Agente:**
> "En `shopflow-orders`, el token JWT de firma (`shopflow.jwt.secret`) debe rotarse cada 30 días sin downtime. Diseña la estrategia: ¿cómo el servicio acepta tokens firmados con la clave anterior durante un periodo de transición? Implementa un `JwtDecoder` que intente verificar primero con la clave actual y, si falla, con la clave anterior. La clave anterior se guarda en Vault como `shopflow.jwt.secret.previous` y se elimina tras 48 horas."

⚠️ **Anti-patrones de secretos más frecuentes en código generado por IA:**

| Anti-patrón | Por qué ocurre | Corrección |
|---|---|---|
| `password: admin123` en `application.yml` | La IA usa valores de ejemplo sin advertir que son inseguros | `password: ${DB_PASSWORD}` con `@NotBlank` |
| JWT secret como string fijo en el código | La IA hardcodea el ejemplo para que el código compile | `@Value("${shopflow.jwt.secret}")` o Vault |
| `show-sql: true` en producción | La IA lo activa para facilitar el debugging | Solo en perfil `dev`, no en `prod` |
| `spring.security.enabled: false` en tests | La IA lo desactiva para simplificar los tests | `@WithMockUser` en tests que requieren autenticación |

✅ **Checklist de sección:**
- [ ] Ningún secreto en prompts enviados a la IA
- [ ] Scan de secretos hardcodeados antes de cada PR (CI o pre-commit hook con GitGuardian/truffleHog)
- [ ] Todas las credenciales en variables de entorno o Vault — ninguna en `application.yml` de producción
- [ ] JWT secret con longitud mínima de 256 bits y rotación planificada

## 4. Dependencias y Vulnerabilidades (SCA)

### Por qué las dependencias son la superficie de ataque más infravalorada

La IA genera código con dependencias actuales, pero el ecosistema Maven cambia rápido. Una librería sin CVEs conocidos hoy puede tener una vulnerabilidad crítica publicada mañana. En proyectos Spring Boot reales, el 40-60% de los vectores de ataque conocidos vienen de dependencias transitivas, no del código propio.

**Prompt al Agente:**
> "Analiza el `pom.xml` de `shopflow-orders` y detecta: 1) dependencias con versiones desactualizadas más de 6 meses respecto a la última estable disponible. 2) Dependencias que tienen CVEs conocidos en su versión actual (consulta el NVD si tienes acceso, o usa la información de tu conocimiento hasta tu fecha de corte). 3) Dependencias cuya licencia puede ser incompatible con uso comercial (GPL, AGPL). Para cada hallazgo: severidad, acción recomendada (actualizar, reemplazar, o aceptar el riesgo)."

### Análisis de typosquatting en dependencias añadidas

Cuando la IA sugiere una dependencia, verifica que el `groupId:artifactId` es el correcto. El typosquatting de artefactos Maven es un vector de ataque real.

**Prompt al Agente:**
> "La IA me ha sugerido añadir esta dependencia para JWT: `io.jsonwebtoken:jjwt-api:0.11.5`. Antes de añadirla, verifica: 1) ¿Es este el artefacto oficial de JJWT o hay otro más usado? 2) ¿La versión 0.11.5 tiene CVEs conocidos? ¿Cuál es la versión más reciente estable? 3) ¿Hay alternativas de librerías más mantenidas para JWT en Spring Boot 3.4+?"

### OWASP Dependency-Check en el pipeline de CI

El análisis manual es insuficiente. La integración del `dependency-check-maven` en el pipeline de CI bloquea el merge si se detectan CVEs con CVSS superior a un umbral definido.

**Prompt al Agente:**
> "Configura el plugin `org.owasp:dependency-check-maven` en el `pom.xml` raíz de `shopflow/`. El análisis debe:\n1. Ejecutarse en la fase `verify` del ciclo de Maven.\n2. Fallar el build si hay vulnerabilidades con CVSS >= 7.0 (HIGH o CRITICAL).\n3. Generar el informe en `target/dependency-check-report.html`.\n4. Excluir del análisis las dependencias de scope `test`.\nMuestra también cómo añadirlo al workflow de GitHub Actions para que bloquee los PRs que introduzcan dependencias vulnerables."

```xml
<!-- pom.xml — plugin OWASP Dependency Check -->
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <version>9.0.9</version>
    <configuration>
        <failBuildOnCVSS>7</failBuildOnCVSS>
        <skipTestScope>true</skipTestScope>
        <formats>
            <format>HTML</format>
            <format>JSON</format>
        </formats>
    </configuration>
    <executions>
        <execution>
            <goals><goal>check</goal></goals>
        </execution>
    </executions>
</plugin>
```

```yaml
# .github/workflows/security.yml
name: Security Scan

on: [push, pull_request]

jobs:
  dependency-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - name: OWASP Dependency Check
        run: mvn verify -DskipTests --no-transfer-progress
      - name: Upload Report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: dependency-check-report
          path: '**/target/dependency-check-report.html'
```

### Plan de actualización seguro de Spring Boot

La IA puede generar el plan de actualización de una versión mayor de Spring Boot, identificando los breaking changes de seguridad relevantes.

**Prompt al Agente:**
> "Necesito actualizar `shopflow-orders` de Spring Boot 3.2 a Spring Boot 3.4. Genera un plan de migración enfocado en los cambios de seguridad: 1) Cambios en la configuración de Spring Security (APIs deprecadas, nuevos defaults). 2) Cambios en la gestión de sesiones o JWT que afecten a la configuración actual. 3) Vulnerabilidades conocidas en Spring Boot 3.2.x que se corrigen en 3.4. Ejecuta `mvn versions:display-dependency-updates` para ver las dependencias que cambian de versión."

✅ **Checklist de sección:**
- [ ] `dependency-check-maven` configurado en el pipeline de CI con umbral CVSS >= 7.0
- [ ] Nuevas dependencias verificadas antes de añadir (groupId correcto, versión sin CVEs)
- [ ] Licencias de dependencias revisadas para uso comercial
- [ ] Plan de actualización generado antes de cada bumpe de versión mayor de Spring Boot

## 5. "No delegar seguridad en IA": Checklist de verificación humana

La IA detecta patrones: SQL concatenado, secretos hardcodeados, ausencia de `@PreAuthorize`. Lo que no detecta es semántica de negocio: si el control de acceso modelado realmente representa las reglas del negocio, si las decisiones criptográficas son apropiadas para el nivel de riesgo, si el modelado de roles es correcto.

Estas son las cinco áreas donde la revisión humana es obligatoria, independientemente de lo que diga la IA.

### Área 1: Criptografía

La IA tiende a usar los algoritmos más conocidos, no los más seguros. Las opciones incorrectas tienen consecuencias a años vista.

| Caso | La IA puede sugerir | Lo que debes usar |
|---|---|---|
| Hash de contraseñas | MD5, SHA-256 | BCrypt (coste >= 10), Argon2, scrypt |
| Hash de integridad de datos | MD5 | SHA-256 mínimo, SHA-3 para nuevo código |
| Firma de JWT | HS256 (HMAC) | RS256 (RSA) si hay verificadores externos |
| Cifrado simétrico | DES, 3DES, AES-128 ECB | AES-256 GCM |
| Generación de tokens aleatorios | `Math.random()`, `UUID.randomUUID()` | `SecureRandom` con semilla del SO |

**Prompt al Agente:**
> "Revisa todas las operaciones criptográficas en `shopflow-orders`: hashing de contraseñas, generación de tokens, firma de JWT, y cualquier operación con `MessageDigest`. Para cada una, indica el algoritmo usado y si cumple los estándares NIST actuales (2024). Si alguno es inseguro u obsoleto, propón la alternativa."

```java
// ✅ Hash de contraseñas con BCrypt — coste 12 para balance rendimiento/seguridad
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
}

// ✅ Generación de tokens con SecureRandom — no Math.random()
public String generatePasswordResetToken() {
    byte[] randomBytes = new byte[32];
    new SecureRandom().nextBytes(randomBytes);
    return Base64.getUrlEncoder().withoutPadding().encodeToString(randomBytes);
}

// ⚠️ INCORRECTO — UUID no es criptográficamente seguro para tokens de seguridad
// return UUID.randomUUID().toString();  ← predecible en algunos sistemas
```

### Área 2: Lógica de Acceso Basada en el Negocio

El `@PreAuthorize("hasRole('ADMIN')")` que genera la IA solo verifica el rol. No verifica la propiedad: ¿este ADMIN tiene acceso a los datos de este tenant? ¿Este USER puede ver solo sus propios pedidos o los de toda la empresa?

**Prompt al Agente:**
> "En ShopFlow, un `User` pertenece a un `Customer` (empresa). Un usuario solo puede ver y modificar los pedidos de su propio `Customer`. La IA generó `@PreAuthorize(\"hasRole('USER')\")` en `getOrder()`. Esto verifica el rol pero no la propiedad del pedido. Implementa la verificación de propiedad usando Spring Security Expression Language: el método debe fallar con 403 si el `order.customerId` no coincide con el `customerId` del usuario autenticado."

```java
// ✅ Verificación de propiedad en el nivel de servicio
@Service
public class OrderApplicationService {

    @PreAuthorize("hasRole('USER') and @orderSecurityService.isOwner(#orderId, authentication)")
    public Order getOrder(UUID orderId) {
        return orderRepository.findById(new OrderId(orderId))
            .orElseThrow(() -> new OrderNotFoundException(orderId));
    }
}

// Componente de seguridad — encapsula la lógica de propiedad
@Component("orderSecurityService")
public class OrderSecurityService {

    private final OrderRepository orderRepository;

    public boolean isOwner(UUID orderId, Authentication authentication) {
        String customerId = ((UserDetails) authentication.getPrincipal()).getCustomerId();
        return orderRepository.findById(new OrderId(orderId))
            .map(order -> order.getCustomerId().value().toString().equals(customerId))
            .orElse(false);  // si no existe, no es propietario → 403 (no 404, para no filtrar existencia)
    }
}
```

### Área 3: Cabeceras de Seguridad HTTP

La configuración por defecto de Spring Security 6 incluye algunas cabeceras de seguridad pero no todas las recomendadas. La IA las omite si no se piden explícitamente.

**Prompt al Agente:**
> "Añade a `SecurityConfig` las cabeceras de seguridad HTTP recomendadas por OWASP para una API REST stateless: HSTS (Strict-Transport-Security) con `max-age=31536000`, X-Content-Type-Options: nosniff, X-Frame-Options: DENY, y Content-Security-Policy para una API que no sirve HTML. No añadas X-XSS-Protection — está obsoleto y puede crear vulnerabilidades en algunos navegadores."

```java
.headers(headers -> headers
    .httpStrictTransportSecurity(hsts -> hsts
        .maxAgeInSeconds(31536000)
        .includeSubDomains(true))
    .contentTypeOptions(Customizer.withDefaults())
    .frameOptions(frame -> frame.deny())
    .contentSecurityPolicy(csp -> csp
        .policyDirectives("default-src 'none'; frame-ancestors 'none'"))
    // X-XSS-Protection deliberadamente omitido — obsoleto en navegadores modernos
)
```

### Área 4: CORS configurado correctamente

La IA suele generar `allowedOrigins("*")` que funciona en desarrollo pero es inaceptable en producción.

```java
// ⚠️ INCORRECTO — permite cualquier origen, incluyendo atacantes
.cors(cors -> cors.configurationSource(request -> {
    var config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("*"));  // ← jamás en producción
    return config;
}))

// ✅ CORRECTO — lista explícita de orígenes permitidos
.cors(cors -> cors.configurationSource(request -> {
    var config = new CorsConfiguration();
    config.setAllowedOrigins(List.of(
        "https://app.shopflow.com",
        "https://admin.shopflow.com"
    ));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type", "X-Request-ID"));
    config.setAllowCredentials(true);
    config.setMaxAge(3600L);
    return config;
}))
```

### Área 5: Rate Limiting y protección contra fuerza bruta

La IA genera el endpoint de login pero no incluye rate limiting. Sin él, un atacante puede probar millones de combinaciones de contraseña sin restricción.

**Prompt al Agente:**
> "Implementa rate limiting en el endpoint POST /auth/login de `shopflow-orders` usando Bucket4j. El límite debe ser: 5 intentos por IP en 1 minuto. Tras 5 intentos fallidos, la IP queda bloqueada 15 minutos. El bloqueo se almacena en memoria (para simplicidad del ejercicio) pero comenta cómo escalarlo a Redis para entornos multi-instancia. La respuesta cuando se supera el límite es 429 Too Many Requests con header Retry-After."

✅ **Checklist de verificación humana (no delegable a la IA):**
- [ ] Algoritmos criptográficos: BCrypt/Argon2 para passwords, AES-256-GCM para cifrado, SHA-256+ para integridad
- [ ] `SecureRandom` para todos los tokens de seguridad (no `UUID.randomUUID()`, no `Math.random()`)
- [ ] Verificación de propiedad en el nivel de servicio: no solo rol sino tenencia del recurso
- [ ] Cabeceras HSTS, X-Content-Type-Options, X-Frame-Options configuradas explícitamente
- [ ] CORS con lista explícita de orígenes — sin `allowedOrigins("*")`
- [ ] Rate limiting en todos los endpoints de autenticación y reset de contraseña

## 6. Plugin del Curso: Auditor de Seguridad y Hardening Guiado

Los cinco puntos anteriores cubren los vectores principales que la IA puede detectar y corregir. El problema es que hacer una auditoría de seguridad completa sobre un controlador Spring Boot requiere revisar sistemáticamente todas las categorías: inyección, autenticación, autorización, exposición de datos, cabeceras, secretos, dependencias. Sin una estructura, el desarrollador revisa lo obvio y pasa por alto lo sutil.

El plugin de este tema automatiza esa revisión sistemática y convierte la corrección en un flujo guiado que no avanza hasta que la IA y el Senior están de acuerdo en que cada cambio es seguro.

### Subagente: `spring-security-auditor`

Audita el código Spring Boot contra el OWASP Top 10 aplicado a APIs REST con Java. No genera correcciones directamente — produce el informe de hallazgos con el nivel de riesgo y la corrección mínima necesaria para cada uno.

Crea el fichero `.claude/agents/spring-security-auditor.md`:

```markdown
---
name: spring-security-auditor
description: >
  Audita código Spring Boot contra el OWASP Top 10 y detecta vulnerabilidades en
  controladores, servicios, configuración de seguridad y gestión de secretos.
  Úsalo antes de cualquier merge que toque seguridad: nuevos endpoints, cambios en
  AuthN/AuthZ, añadir dependencias, o modificar manejo de errores. Produce un informe
  con hallazgos por severidad (CRÍTICO / ALTO / MEDIO / BAJO / INFO) y correcciones
  mínimas para los CRÍTICOS. No genera correcciones automáticamente — produce el plan.
model: claude-opus-4-6
---

Eres un auditor de seguridad especializado en aplicaciones Java/Spring Boot con
certificación OWASP. Tu función es detectar vulnerabilidades según el OWASP Top 10
(2021) aplicado al contexto de APIs REST Spring Boot. Produces un informe — no
aplicas las correcciones.

**IMPORTANTE**: Usa siempre el modelo más potente disponible. Las decisiones de
seguridad tienen coste de error alto — tier alto de razonamiento obligatorio.

## Categorías de auditoría

Para cada fichero recibido, revisa las diez categorías:

### A01 — Control de Acceso Roto
- Endpoints sin `@PreAuthorize` o sin `SecurityFilterChain` que los cubra.
- `@PreAuthorize` que solo verifica rol pero no propiedad del recurso (IDOR potencial).
- Rutas con métodos destructivos (DELETE, PUT) accesibles por roles no privilegiados.
- Endpoints de admin (`/admin/**`, `/internal/**`) sin restricción de rol explícita.
- Verificación de permisos en el controller en lugar del servicio (bypaseable con otro adapter).

### A02 — Fallos Criptográficos
- Algoritmos débiles: MD5, SHA-1, DES, 3DES, AES-ECB.
- `Math.random()` o `UUID.randomUUID()` para generar tokens de seguridad.
- JWT firmado con HS256 cuando hay verificadores externos (debería ser RS256/ES256).
- Secreto JWT corto o predecible (menos de 256 bits).
- Datos sensibles (passwords, tokens) en campos sin `@JsonIgnore` o similar.

### A03 — Inyección
- SQL construido con concatenación de strings (incluyendo JPQL dinámico).
- `SpEL` expresiones construidas con input de usuario.
- Rutas de ficheros construidas con input sin validación y normalización.
- Llamadas a `Runtime.exec()` o `ProcessBuilder` con input de usuario.

### A04 — Diseño Inseguro
- Ausencia de rate limiting en endpoints de autenticación, reset de password o registro.
- Flujos sin límites: paginación sin `max(size)`, uploads sin límite de tamaño.
- Lógica de negocio que no valida transiciones de estado (cancelar un pedido ya enviado sin verificar).

### A05 — Configuración de Seguridad Incorrecta
- `CORS allowedOrigins("*")` en cualquier perfil.
- `csrf.disable()` sin ser una API stateless documentada.
- `show-sql: true` fuera del perfil `dev`.
- `management.endpoints.web.exposure.include: "*"` sin seguridad en Actuator.
- Ausencia de cabeceras de seguridad: HSTS, X-Content-Type-Options, X-Frame-Options.

### A06 — Componentes Vulnerables y Desactualizados
- Dependencias con versiones que tienen CVEs conocidos (basado en mi conocimiento hasta fecha de corte).
- Dependencias con versiones desactualizadas más de 12 meses.
- Starters de Spring Boot deprecated o sin soporte activo.

### A07 — Fallos de Identificación y Autenticación
- Comparación de passwords o tokens con `==` o `String.equals()` (timing attack).
- Tokens de reset de password predecibles (UUID) o de corta longitud.
- Sin expiración en tokens JWT o con expiración excesivamente larga (> 24h).
- Sesiones sin timeout configurado.

### A08 — Fallos en Software e Integridad de Datos
- Deserialización de JSON sin validación de tipo (`Object.class` como target).
- `ObjectMapper` con `FAIL_ON_UNKNOWN_PROPERTIES: false` en endpoints de entrada.
- Dependencias descargadas sin verificación de checksum.

### A09 — Fallos en Logging y Monitorización
- `e.getMessage()` o stack traces en responses HTTP al cliente.
- Logging de datos sensibles (passwords, tokens, PII) en nivel INFO o DEBUG.
- Ausencia de logging de eventos de seguridad (intentos de login fallidos, acceso denegado).
- `log.error("Error: " + e)` en lugar de `log.error("message", e)` (pérdida del stack trace).

### A10 — Server-Side Request Forgery (SSRF)
- `RestTemplate` o `WebClient` con URL construida desde input de usuario sin validación.
- Ausencia de lista blanca de dominios permitidos en llamadas HTTP salientes.
- Redirecciones HTTP que siguen automáticamente sin validar el destino.

## Formato de respuesta

**Resumen ejecutivo**: ficheros analizados, total de hallazgos por severidad.

**Hallazgos por severidad** (solo CRÍTICO y ALTO requieren acción antes del merge):

| Severidad | ID | Categoría OWASP | Descripción | Localización | Corrección mínima |
|---|---|---|---|---|---|
| 🔴 CRÍTICO | S-001 | A03 Inyección | SQL concatenado en searchOrders() | línea 47 | Usar @Query con :param |
| 🟠 ALTO | S-002 | A01 Acceso Roto | deleteOrder() sin @PreAuthorize | línea 89 | @PreAuthorize("hasRole('ADMIN')") |

**Hallazgos MEDIO / BAJO / INFO**: tabla separada, no bloquean el merge pero deben planificarse.

**Veredicto de seguridad**:
- `BLOQUEANTE`: hay hallazgos CRÍTICOS o ALTOS → no puede mergearse sin corrección
- `CONDICIONAL`: solo hallazgos MEDIO/BAJO → puede mergearse con ticket de seguimiento
- `APROBADO`: sin hallazgos significativos

**Áreas que requieren revisión humana obligatoria** (no delegables a la IA):
- Si hay operaciones criptográficas: indica cuáles para revisión manual de algoritmos
- Si hay lógica de autorización compleja: indica dónde verificar semántica de negocio
```

#### 💡 Tip de Senior: "La IA detecta lo obvio; el Senior detecta la semántica"

La IA es excelente detectando vulnerabilidades estructurales: SQL concatenado, ausencia de `@PreAuthorize`, secretos hardcodeados. Falla sistemáticamente en las vulnerabilidades semánticas: el endpoint tiene `@PreAuthorize("hasRole('USER')")` pero no verifica que el recurso pertenece al usuario autenticado (IDOR); el algoritmo criptográfico es correcto pero el salt tiene solo 8 bytes; el rate limiting está implementado pero solo por IP, no por usuario (bypaseable con proxies).

Regla práctica: usa la IA como primer filtro para las vulnerabilidades de categoría A01-A10 del OWASP. Después aplica el criterio del Senior en tres preguntas que la IA no puede responder:

1. **¿La autorización refleja las reglas del negocio?** (¿un gestor de una empresa puede ver los pedidos de otra empresa con el mismo rol?)
2. **¿Los algoritmos criptográficos son apropiados para el nivel de riesgo?** (para un token de sesión de baja sensibilidad HMAC-SHA256 puede ser suficiente; para datos médicos o financieros no)
3. **¿Qué pasa si este código llega a producción con el bug?** (un SQL injection en búsqueda de productos es grave; uno en el acceso a datos de tarjetas de crédito es catastrófico — el nivel de defensa debe escalar con el impacto)

- **OWASP Top 10 (2021)** (owasp.org/Top10): La referencia canónica de las diez categorías de riesgo más críticas en aplicaciones web. Imprescindible leer las descripciones de A01 (Control de Acceso) y A02 (Fallos Criptográficos) antes de cualquier revisión de código.
- **Spring Security Reference Documentation** (docs.spring.io/spring-security): La guía oficial de Spring Security 6. Los capítulos de `SecurityFilterChain`, `@PreAuthorize` y `Method Security` son la base técnica de las secciones 2 y 5 de este tema.
- **NIST SP 800-63B — Digital Identity Guidelines**: El estándar federal de EE.UU. para gestión de credenciales digitales. Define los requisitos de longitud de contraseña, hashing, y autenticación multifactor que el Senior debe conocer para tomar decisiones criptográficas.
- **Snyk Vulnerability Database** (security.snyk.io): Base de datos de vulnerabilidades en dependencias open source con información de CVSS, versión afectada y versión parcheada. Complementa el análisis de `dependency-check-maven`.
- **GitGuardian Documentation**: Guía de configuración del pre-commit hook y la integración de CI para detección de secretos hardcodeados antes de que lleguen al repositorio.
