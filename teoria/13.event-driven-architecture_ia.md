# Tema 13. EVENT-DRIVEN ARCHITECTURE CON COPILOT: DE "DEMO" A PRODUCCIÓN

La arquitectura orientada a eventos (EDA) introduce una complejidad que la IA a menudo ignora: el fallo parcial y la asincronía. Cuando el agente genera un publicador de Kafka, el resultado predeterminado es `kafkaTemplate.send(topic, payload)` dentro de un `@Transactional`. Eso compila, los tests unitarios con mocks pasan, y la demo funciona en local. En producción, el contenedor de Kafka se reinicia, la base de datos confirma la orden, el evento nunca llega y el cliente no recibe su confirmación. En este tema aprenderás a guiar al agente para que genere sistemas de mensajería robustos, tolerantes a fallos y observables desde el primer commit.

---

## 1. Definición de eventos, schemas y versionado

### Eventos de Dominio vs. Eventos de Integración

El primer error que comete el agente es mezclar los dos tipos de eventos. Debes forzar la distinción desde el prompt:

- **Eventos de Dominio** (`domain/event/`): internos al bounded context, inmutables, representan algo que ocurrió en el dominio. Solo el propio módulo los publica. Sin dependencias de framework, Java puro.
- **Eventos de Integración** (`infrastructure/messaging/`): serializados y publicados al broker. Adaptan el evento de dominio al contrato externo (Avro, JSON+schema). Pueden tener campos extra para consumidores externos (versión, correlationId, timestamp ISO-8601).

```java
// domain/event/OrderCreatedEvent.java — Java puro, sin framework
public record OrderCreatedEvent(
    String orderId,
    String customerId,
    BigDecimal totalAmount,
    Instant occurredAt
) {}

// infrastructure/messaging/OrderCreatedIntegrationEvent.java — contrato externo serializable
public record OrderCreatedIntegrationEvent(
    String eventId,       // UUID para idempotencia
    String eventType,     // "order.created"
    String version,       // "1.0"
    String orderId,
    String customerId,
    String totalAmount,   // String para evitar problemas de precisión en JSON
    String currency,
    Instant occurredAt
) {
    public static OrderCreatedIntegrationEvent from(OrderCreatedEvent domain) {
        return new OrderCreatedIntegrationEvent(
            UUID.randomUUID().toString(),
            "order.created",
            "1.0",
            domain.orderId(),
            domain.customerId(),
            domain.totalAmount().toPlainString(),
            "EUR",
            domain.occurredAt()
        );
    }
}
```

**Prompt al Agente:**
> "En `shopflow-orders`, tengo el evento de dominio `OrderCreatedEvent` en `domain/event/`. Crea el evento de integración `OrderCreatedIntegrationEvent` en `infrastructure/messaging/event/`. El evento de integración debe añadir `eventId` (UUID), `eventType` (string), `version` ("1.0"), `currency` ("EUR") y un método estático `from(OrderCreatedEvent)`. El campo `totalAmount` debe ser `String` (no `BigDecimal`) para evitar problemas de precisión en JSON. Sin anotaciones de Jackson en el `record` de dominio."

## 2. Publicación y Consumo con garantías reales

### El Anti-patrón del Dual-Write

Este es el antipatrón más peligroso que el agente genera por defecto:

```java
// ⚠️ ANTI-PATRÓN: dual-write sin garantía de atomicidad
@Transactional
public Order createOrder(CreateOrderCommand cmd) {
    Order order = Order.create(cmd);
    orderRepository.save(order);                            // ① persiste en DB
    kafkaTemplate.send("orders.created", toEvent(order));  // ② publica en Kafka
    return order;
}
```

El problema: si Kafka no está disponible en el momento del paso ②, `kafkaTemplate.send()` lanza una excepción, la transacción hace rollback en la base de datos — pero si el send ya entró en el buffer de Kafka antes del error, el evento puede haberse enviado aunque la orden no exista en DB. El camino contrario también falla: si la DB confirma (paso ①) y Kafka falla (paso ②), la orden existe pero el evento nunca llega.

### Outbox Pattern: la solución canónica

La solución es guardar el evento en la misma transacción de DB que la entidad, y publicarlo en un paso separado con reintento:

```java
// infrastructure/persistence/OutboxEvent.java
@Entity
@Table(name = "outbox_events")
public class OutboxEvent {

    @Id
    private String eventId;

    @Column(nullable = false)
    private String topic;

    @Column(nullable = false)
    private String eventType;

    @Column(columnDefinition = "TEXT", nullable = false)
    private String payload;    // JSON serializado

    @Column(nullable = false)
    private Instant createdAt;

    @Column
    private Instant publishedAt;  // null = pendiente de publicar

    public static OutboxEvent of(String topic, String eventType, String payload) {
        var event = new OutboxEvent();
        event.eventId = UUID.randomUUID().toString();
        event.topic = topic;
        event.eventType = eventType;
        event.payload = payload;
        event.createdAt = Instant.now();
        return event;
    }

    public void markPublished() {
        this.publishedAt = Instant.now();
    }

    public boolean isPending() {
        return publishedAt == null;
    }
}
```

```java
// application/service/OrderApplicationService.java — usa el Outbox
@Service
@RequiredArgsConstructor
public class OrderApplicationService {

    private final OrderRepository orderRepository;
    private final OutboxEventRepository outboxEventRepository;
    private final ObjectMapper objectMapper;

    @Transactional  // ← una sola transacción cubre DB + outbox
    public Order createOrder(CreateOrderCommand cmd) {
        Order order = Order.create(cmd);
        orderRepository.save(order);

        // Persiste el evento en la misma transacción — atomicidad garantizada
        OrderCreatedIntegrationEvent integrationEvent =
            OrderCreatedIntegrationEvent.from(order.domainEvent());
        outboxEventRepository.save(OutboxEvent.of(
            "shopflow.orders.created",
            "order.created",
            objectMapper.writeValueAsString(integrationEvent)
        ));

        return order;
    }
}
```

```java
// infrastructure/messaging/OutboxRelayJob.java — relay en paso separado
@Component
@RequiredArgsConstructor
@Slf4j
public class OutboxRelayJob {

    private final OutboxEventRepository outboxEventRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 5_000)  // cada 5 segundos
    @Transactional
    public void relay() {
        List<OutboxEvent> pending = outboxEventRepository.findByPublishedAtIsNull();
        for (OutboxEvent event : pending) {
            try {
                kafkaTemplate.send(event.getTopic(), event.getEventId(), event.getPayload())
                    .get(2, TimeUnit.SECONDS);  // confirma que Kafka recibió el mensaje
                event.markPublished();
                outboxEventRepository.save(event);
                log.info("Outbox: published eventId={} topic={}", event.getEventId(), event.getTopic());
            } catch (Exception e) {
                log.warn("Outbox: failed to publish eventId={}, will retry", event.getEventId(), e);
                // No hace nada — el siguiente ciclo lo reintentará
            }
        }
    }
}
```

**Prompt al Agente:**
> "En `shopflow-orders`, implementa el Outbox Pattern para `OrderApplicationService.createOrder()`. Fase 1: crea la entidad JPA `OutboxEvent` en `infrastructure/persistence/` con campos `eventId` (UUID), `topic`, `eventType`, `payload` (TEXT), `createdAt`, `publishedAt` (nullable). Crea también `OutboxEventRepository`. Fase 2: crea `OutboxRelayJob` en `infrastructure/messaging/` con un `@Scheduled` cada 5 segundos que lea eventos con `publishedAt IS NULL`, los publique con `kafkaTemplate.send().get(2, SECONDS)` y actualice `publishedAt` solo si Kafka confirma. Sin modificar el método `createOrder()` todavía — solo las clases de infraestructura."

### Consumo Resiliente: DefaultErrorHandler + DLQ

El agente genera consumidores que hacen `log.error(e)` y continúan. En producción, un mensaje malformado o que siempre falla bloquearía el procesamiento de todos los siguientes en esa partición:

```java
// ⚠️ ANTI-PATRÓN: consumidor sin manejo de errores
@KafkaListener(topics = "shopflow.orders.created")
public void onOrderCreated(String payload) {
    var event = objectMapper.readValue(payload, OrderCreatedIntegrationEvent.class);
    notificationService.sendOrderConfirmation(event);
    // Si notificationService lanza excepción: el mensaje se reintentar infinitamente
    // o se pierde si hay auto-commit
}
```

La configuración correcta con `DefaultErrorHandler` y DLQ:

```java
// infrastructure/config/KafkaConsumerConfig.java
@Configuration
@RequiredArgsConstructor
public class KafkaConsumerConfig {

    private final KafkaTemplate<String, String> kafkaTemplate;

    @Bean
    public DefaultErrorHandler errorHandler() {
        // 3 reintentos con backoff: 1s, 2s, 4s
        var backOff = new ExponentialBackOffWithMaxRetries(3);
        backOff.setInitialInterval(1_000);
        backOff.setMultiplier(2.0);
        backOff.setMaxInterval(10_000);

        // Dead Letter Topic: shopflow.orders.created.DLT
        var dlqRecoverer = new DeadLetterPublishingRecoverer(kafkaTemplate,
            (record, ex) -> new TopicPartition(record.topic() + ".DLT", record.partition()));

        var handler = new DefaultErrorHandler(dlqRecoverer, backOff);

        // Errores de deserialización no se reintentan — van directo al DLT
        handler.addNotRetryableExceptions(JsonProcessingException.class);

        return handler;
    }
}
```

```java
// infrastructure/messaging/OrderCreatedConsumer.java — consumidor robusto
@Component
@RequiredArgsConstructor
@Slf4j
public class OrderCreatedConsumer {

    private final NotificationService notificationService;
    private final ProcessedEventRepository processedEventRepository;

    @KafkaListener(
        topics = "shopflow.orders.created",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void onOrderCreated(
            @Payload String payload,
            @Header(KafkaHeaders.RECEIVED_KEY) String eventId,
            Acknowledgment ack) {

        // Idempotencia: ignorar si ya procesamos este eventId
        if (processedEventRepository.existsByEventId(eventId)) {
            log.info("Skipping duplicate event eventId={}", eventId);
            ack.acknowledge();
            return;
        }

        try {
            var event = objectMapper.readValue(payload, OrderCreatedIntegrationEvent.class);
            notificationService.sendOrderConfirmation(event.orderId(), event.customerId());
            processedEventRepository.save(new ProcessedEvent(eventId, Instant.now()));
            ack.acknowledge();  // ACK manual: solo si procesamiento fue exitoso
            log.info("Processed order.created eventId={} orderId={}", eventId, event.orderId());
        } catch (JsonProcessingException e) {
            log.error("Deserialize failed for eventId={}, sending to DLT", eventId, e);
            ack.acknowledge();  // mensaje malformado: no reintentar, el DefaultErrorHandler lo enruta al DLT
        }
    }
}
```

**Prompt al Agente:**
> "Configura `DefaultErrorHandler` en `KafkaConsumerConfig` dentro de `shopflow-notifications` con: 3 reintentos con `ExponentialBackOffWithMaxRetries`, backoff inicial 1s, multiplicador 2, y `DeadLetterPublishingRecoverer` que envía al tópico `shopflow.orders.created.DLT`. `JsonProcessingException` debe ser no-reintentable. Luego actualiza `OrderCreatedConsumer` para usar ACK manual (`Acknowledgment ack`) e idempotencia: antes de procesar, verifica que `eventId` no exista en la tabla `processed_events`. Solo hace `ack.acknowledge()` si el procesamiento fue exitoso o si el mensaje era un duplicado."

✅ **Checklist de sección:**
- [ ] Outbox Pattern en lugar de `kafkaTemplate.send()` dentro de `@Transactional`
- [ ] `OutboxRelayJob` con `get(2, SECONDS)` — confirmación explícita de Kafka
- [ ] `DefaultErrorHandler` con `ExponentialBackOffWithMaxRetries` y DLQ configurado
- [ ] ACK manual: `AckMode.MANUAL_IMMEDIATE` en el container factory
- [ ] Idempotencia en el consumidor: tabla `processed_events` o Redis con TTL
- [ ] `JsonProcessingException` en la lista de excepciones no reintentables

## 3. Consistencia Eventual: Casos típicos y Testing

### Compensación en Sagas

Cuando una saga falla en el paso N, los pasos 1..N-1 ya han persistido su estado. La IA puede ayudarte a diseñar la cadena de compensación, pero necesitas darle el contexto completo del flujo:

**Prompt al Agente:**
> "En ShopFlow, el flujo de creación de un pedido tiene estos pasos: 1) `Order.confirm()` en DB, 2) reserva de inventario vía evento, 3) cargo al método de pago. Si el paso 3 falla, el inventario ya está reservado y la orden está confirmada. Diseña los eventos de compensación necesarios: `PaymentFailedEvent` que desencadene `InventoryReservationReleasedEvent` y `OrderCancelledEvent`. Genera los handlers de compensación en `shopflow-orders` y `shopflow-notifications`. Sin Saga Orchestrator — usa el patrón Choreography con los eventos que ya tenemos."

### Testing de Asincronía con Awaitility

Nunca uses `Thread.sleep()` en tests de EDA. La duración exacta de procesamiento es no determinista y varía entre entornos:

```java
// ⚠️ ANTI-PATRÓN: sleep en tests de eventos
@Test
void orderCreated_notificationSent() throws InterruptedException {
    createOrder(validCommand());
    Thread.sleep(3000);  // ← frágil: falla en CI, demasiado lento en local
    assertThat(notificationRepository.count()).isEqualTo(1);
}
```

```java
// ✅ Con Awaitility + Testcontainers Kafka
@SpringBootTest
@Testcontainers
class OrderNotificationIntegrationTest {

    @Container
    static KafkaContainer kafka = new KafkaContainer(
        DockerImageName.parse("confluentinc/cp-kafka:7.5.0"));

    @DynamicPropertySource
    static void kafkaProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.kafka.bootstrap-servers", kafka::getBootstrapServers);
    }

    @Autowired OrderApplicationService orderService;
    @Autowired NotificationRepository notificationRepository;

    @Test
    void whenOrderCreated_thenNotificationPersisted() {
        var cmd = new CreateOrderCommand("customer-1", List.of(
            new OrderItemCommand("product-1", 2, new BigDecimal("49.99"))
        ));

        orderService.createOrder(cmd);

        // Awaitility espera hasta 10 segundos con polling cada 500ms
        await()
            .atMost(10, SECONDS)
            .pollInterval(500, MILLISECONDS)
            .untilAsserted(() ->
                assertThat(notificationRepository.findByCustomerId("customer-1"))
                    .isNotEmpty()
                    .first()
                    .extracting(Notification::getType)
                    .isEqualTo("ORDER_CONFIRMATION")
            );
    }
}
```

**Prompt al Agente:**
> "Crea el test de integración `OrderNotificationIntegrationTest` en `shopflow-notifications`. Usa `@Testcontainers` con `KafkaContainer` (imagen `confluentinc/cp-kafka:7.5.0`), `@DynamicPropertySource` para inyectar el bootstrap server, y Awaitility para esperar hasta 10 segundos con polling cada 500ms. El test debe: 1) publicar un `OrderCreatedIntegrationEvent` directamente en el tópico `shopflow.orders.created` con un `KafkaTemplate` de test, 2) esperar con `await().untilAsserted()` que aparezca una `Notification` en la base de datos. Sin `Thread.sleep()`."

## 4. Tests de Contrato, Consumidor y Productor

### Testcontainers Kafka: test de integración real

La regla es simple: si el consumidor procesa eventos de Kafka, el test de integración debe usar Kafka real (Testcontainers), no un mock. Los mocks de KafkaTemplate verifican que se llama al método, no que el mensaje llega y es procesado correctamente con la configuración de serialización real.

```java
// test de integración del productor — verifica que el evento llega al tópico correcto
@SpringBootTest
@Testcontainers
class OutboxRelayJobIntegrationTest {

    @Container
    static KafkaContainer kafka = new KafkaContainer(
        DockerImageName.parse("confluentinc/cp-kafka:7.5.0"));

    @Autowired OutboxEventRepository outboxEventRepository;
    @Autowired OutboxRelayJob relayJob;

    @Test
    void pendingOutboxEvent_isPublishedToKafka() throws Exception {
        // Arrange: evento pendiente en outbox
        outboxEventRepository.save(OutboxEvent.of(
            "shopflow.orders.created",
            "order.created",
            """{"eventId":"evt-1","orderId":"ord-1","customerId":"cust-1"}"""
        ));

        // Act: dispara el relay manualmente (sin esperar el @Scheduled)
        relayJob.relay();

        // Assert: el evento fue marcado como publicado
        assertThat(outboxEventRepository.findByEventId("evt-1"))
            .isPresent()
            .get()
            .extracting(OutboxEvent::getPublishedAt)
            .isNotNull();
    }
}
```

**Prompt al Agente:**
> "Crea el test de integración `OutboxRelayJobIntegrationTest` usando `@Testcontainers` con `KafkaContainer`. El test debe: 1) insertar un `OutboxEvent` pendiente en la base de datos, 2) invocar `relayJob.relay()` directamente, 3) verificar con AssertJ que el `OutboxEvent` tiene `publishedAt != null`. Usa `@DynamicPropertySource` para configurar el bootstrap server. Sin mocks de `KafkaTemplate`."

### Cuándo usar mocks vs. infraestructura real

| Tipo de test | Qué verificar | Estrategia recomendada |
|---|---|---|
| Lógica de negocio del consumidor | Que `notificationService.send()` se llama con los datos correctos | Mock — el mensaje ya llegó, solo prueba lo que hace con él |
| Serialización del evento | Que el JSON producido por el productor es parseable por el consumidor | Testcontainers Kafka — prueba el round-trip real |
| `DefaultErrorHandler` y DLQ | Que un mensaje que falla N veces llega al DLT | Testcontainers Kafka — la lógica de retry vive en la configuración del container |
| Outbox relay | Que el evento pendiente se publica y se marca | Testcontainers Kafka + DB real |
| Tests unitarios de transformación (mapper) | Que `OrderCreatedIntegrationEvent.from()` mapea bien | Test unitario puro sin Kafka |

**Prompt al Agente:**
> "Tengo el test de unidad `OrderCreatedConsumerTest` que mockea `KafkaTemplate`. Quiero añadir un test de integración que verifique que un `JsonProcessingException` (payload malformado) enruta el mensaje al tópico `shopflow.orders.created.DLT` sin reintentos. Usa Testcontainers Kafka con el `KafkaConsumerConfig` real (no mocks). El test debe verificar que hay un mensaje en el DLT con el `eventId` del mensaje original."

- [ ] Test explícito del comportamiento del DLT con mensajes malformados

## 5. Observabilidad de Eventos: Tracing y Correlation IDs

### Propagación del contexto de tracing a través de Kafka

Micrometer Tracing propaga el contexto de tracing en headers HTTP automáticamente. En Kafka no es automático: debes configurar la propagación explícitamente.

```yaml
# application.yml — activar propagación de tracing en Kafka
spring:
  kafka:
    producer:
      properties:
        spring.instrumentation.messaging.kafka.enabled: true
    consumer:
      properties:
        spring.instrumentation.messaging.kafka.enabled: true
management:
  tracing:
    sampling:
      probability: 1.0   # 100% en desarrollo; ajustar en producción
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans
```

Cuando esto está correctamente configurado, el `traceId` del request HTTP que crea la orden viaja en el header `traceparent` del mensaje Kafka y el consumidor en `shopflow-notifications` lo recoge, enlazando los spans del productor y consumidor en Zipkin o Grafana Tempo.

**Prompt al Agente:**
> "Configura Micrometer Tracing con propagación W3C TraceContext en `shopflow-orders` y `shopflow-notifications`. En el productor, asegúrate de que el `traceId` actual viaja en el header `traceparent` del mensaje Kafka. En el consumidor, extrae el `traceId` del header y establécelo en el `MDC` antes de procesar el mensaje, para que aparezca en todos los logs del consumidor."

### Logging estructurado en consumidores

El log del consumidor debe incluir siempre: el `traceId`, el `eventId`, el `orderId` y el tipo de evento. Sin esto, depurar un error en producción con millones de mensajes es inviable:

```java
// ✅ Log estructurado en el consumidor
@KafkaListener(topics = "shopflow.orders.created")
public void onOrderCreated(
        @Payload String payload,
        @Header(KafkaHeaders.RECEIVED_KEY) String eventId,
        @Header(value = "traceparent", required = false) String traceparent,
        Acknowledgment ack) {

    // Establece el contexto en MDC para todos los logs de este mensaje
    MDC.put("eventId", eventId);
    MDC.put("traceparent", traceparent);

    try {
        var event = objectMapper.readValue(payload, OrderCreatedIntegrationEvent.class);
        MDC.put("orderId", event.orderId());
        MDC.put("customerId", event.customerId());

        log.info("Processing event type={} orderId={}", event.eventType(), event.orderId());
        notificationService.sendOrderConfirmation(event.orderId(), event.customerId());
        ack.acknowledge();
        log.info("Completed event orderId={}", event.orderId());

    } catch (Exception e) {
        log.error("Failed processing event eventId={}", eventId, e);
        throw e;  // deja que DefaultErrorHandler gestione el retry
    } finally {
        MDC.clear();  // limpia el MDC — el thread vuelve al pool
    }
}
```

### Métricas de Consumer Lag

El consumer lag (diferencia entre el último offset producido y el último offset consumido) es la métrica más importante de un sistema Kafka. Si el lag crece de forma sostenida, el consumidor no puede seguir el ritmo del productor.

**Prompt al Agente:**
> "Añade un `MeterRegistry` en `OrderCreatedConsumer` para registrar dos métricas: 1) `kafka.consumer.processed.total` (contador por `eventType` y resultado `success`/`error`), 2) `kafka.consumer.processing.duration` (timer por `eventType`). Usa `@Timed` de Micrometer si es aplicable al contexto del listener, o `Timer.Sample` manual si no lo es. Asegúrate de que las métricas incluyen el tag `application: shopflow-notifications`."

⚠️ **Anti-patrones de observabilidad:**

```java
// ⚠️ Log sin contexto: inútil en producción
log.info("Procesando evento");                     // ¿Qué evento? ¿De qué orden?

// ⚠️ Log del stack trace como toString: dificulta el parsing
log.error("Error: " + e.toString());               // Usa log.error("msg", e) siempre

// ⚠️ MDC sin finally: el contexto se contamina entre mensajes
MDC.put("orderId", orderId);
process();                                          // Si lanza excepción, el MDC nunca se limpia
// → el próximo mensaje del thread hereda el orderId anterior
```

✅ **Checklist de sección:**
- [ ] `spring.instrumentation.messaging.kafka.enabled: true` en productor y consumidor
- [ ] Header `traceparent` propagado en cada mensaje Kafka
- [ ] `MDC.put("eventId", ...)` y `MDC.put("orderId", ...)` al inicio del listener
- [ ] `MDC.clear()` en bloque `finally` — nunca fuera de él
- [ ] Métricas de procesamiento registradas con `MeterRegistry`
- [ ] Alert de consumer lag configurada en el broker (Confluent Control Center o Prometheus Kafka Exporter)

## 6. Plugin del Curso: Revisor de Eventos y Añadir Publicación de Eventos

En arquitecturas EDA, los errores más costosos en producción tienen un patrón común: un desarrollador añade un nuevo caso de uso (cancelar pedido, procesar devolución) y olvida alguna de las tres garantías críticas — atomicidad (Outbox), resiliencia (DLQ) o observabilidad (traceId + log estructurado). El plugin de este tema automatiza dos cosas: revisar que el código EDA existente cumple los estándares de producción, y generar el scaffolding completo para publicar un nuevo evento con todas las garantías desde el primer commit.

### Subagente: `spring-event-reviewer`

Auditor especializado en código Kafka/EDA. Lo usas cada vez que alguien añade o modifica un productor, consumidor, o configuración de mensajería.

Crea el fichero `.claude/agents/spring-event-reviewer.md`:

## Categorías de Auditoría

### 1. Atomicidad del Productor
- **CRÍTICO**: `kafkaTemplate.send()` dentro de un método `@Transactional` sin Outbox Pattern.
- **CRÍTICO**: Ausencia de `OutboxEvent` o tabla `outbox_events` — publicación directa sin relay.
- **ADVERTENCIA**: `OutboxRelayJob` sin timeout en `kafkaTemplate.send().get()` — puede bloquearse indefinidamente.
- **ADVERTENCIA**: Relay sin backoff — reintentos inmediatos saturan Kafka durante un incidente.

### 2. Resiliencia del Consumidor
- **CRÍTICO**: Ausencia de `DefaultErrorHandler` con DLQ — mensajes que fallan se pierden o bloquean la partición.
- **CRÍTICO**: ACK automático (modo por defecto) sin ACK manual — mensaje marcado como consumido antes de procesarlo.
- **ADVERTENCIA**: `JsonProcessingException` no está en la lista de errores no-reintentables — mensajes malformados consumen todos los reintentos.
- **ADVERTENCIA**: Sin tabla `processed_events` ni check de idempotencia — el mismo mensaje se procesa dos veces en caso de redelivery.

### 3. Schema y Contrato
- **ADVERTENCIA**: Evento de integración sin campo `eventId` (UUID) — imposible deduplicar en el consumidor.
- **ADVERTENCIA**: Evento de integración sin campo `version` — imposible gestionar compatibilidad al evolucionar el schema.
- **ADVERTENCIA**: Tipo `double` o `float` en campos de dinero — usar `String` o `BigDecimal`.
- **INFO**: Evento de dominio (`domain/event/`) mezcla con clases de framework (anotaciones Jackson, Kafka) — violar la pureza del dominio.

### 4. Observabilidad
- **ADVERTENCIA**: Listener sin `MDC.put("eventId", ...)` al inicio — trazabilidad imposible en producción.
- **ADVERTENCIA**: `MDC.put()` sin `MDC.clear()` en bloque `finally` — contaminación del contexto entre mensajes.
- **ADVERTENCIA**: Sin propagación de `traceparent` entre productor y consumidor — spans desconectados en Zipkin/Tempo.
- **INFO**: Log de inicio del procesamiento sin `orderId` u otro identificador de negocio.

### 5. Testing
- **ADVERTENCIA**: Tests del consumidor usan solo mocks de `KafkaTemplate` — la configuración de retry/DLQ no se verifica.
- **ADVERTENCIA**: Tests con `Thread.sleep()` en lugar de Awaitility — frágiles en CI.
- **INFO**: Sin test que verifique el comportamiento del DLT con mensajes malformados.

#### 💡 Tip de Senior

El Outbox Pattern tiene un coste operacional que la IA no menciona: la tabla `outbox_events` crece indefinidamente si no hay un job de limpieza. Añade siempre un `@Scheduled` semanal que elimine los eventos con `published_at` anterior a 7 días. En sistemas de alto tráfico, esta tabla puede generar millones de filas en horas.

Otro punto que la IA ignora sistemáticamente: el relay job con `@Scheduled(fixedDelay = 5_000)` en múltiples instancias del mismo servicio puede publicar el mismo evento dos veces (una por instancia). La solución no es serializar las instancias — es garantizar que el consumidor sea idempotente, que es exactamente lo que la tabla `processed_events` resuelve.
